   250→### UC-12: 매체 활성/비활성 전환
   251→
   252→**누가**: 어드민, 매체사 또는 운영대행사
   253→**언제**: 매체 운영을 일시 중지하거나 재개할 때
   254→**전제조건**: UC-07로 매체 등록 완료
   255→
   256→#### 시나리오
   257→
   258→1. 어드민, 매체사 또는 운영대행사가 매체 관리 > 목록에서 대상 매체를 선택한다
   259→2. 매체 상세에서 "비활성화" 버튼을 클릭한다
   260→3. 시스템이 확인 모달을 표시한다: "이 매체를 비활성화하시겠습니까? 진행 중인 편성표가 일시정지됩니다. 편성표 종료일이 경과하면 자동 종료됩니다."
   261→4. 확인 버튼을 누르면 매체 상태가 "비활성"으로 변경된다
   262→
   263→#### 이런 경우에는
   264→
   265→- **진행 중인 편성표가 있는 매체 비활성화**: "이 매체에 적용중인 편성표가 N건 있습니다. 비활성화하면 모두 일시정지됩니다. 종료일이 경과한 편성표는 자동 종료됩니다. 계속하시겠습니까?" 경고
   266→- **진행 중인 캠페인이 연결된 매체 비활성화**: "이 매체에 집행 중인 캠페인이 N건 있습니다." 추가 경고
   267→- **비활성 매체를 다시 활성화**: "활성화" 버튼 클릭 → 즉시 활성화. `일시정지` 상태의 편성표 중 종료일이 경과하지 않은 것은 자동으로 `적용중`으로 복귀. 종료일이 이미 경과한 편성표는 `종료` 처리 (재편성 필요)
   268→- **매체 비활성화 시 편성표 상태 경합**: 비활성화/활성화와 편성표 상태 전이를 원자적으로 처리한다
   269→  - 비활성화: 해당 매체의 모든 "적용중" 편성표를 "일시정지"로 전환 (단일 트랜잭션)
   270→  - 활성화: 현재 시각 기준으로 종료일을 재검사한 후 상태 전환 (종료일 경과 → "종료", 미경과 → "적용중")
   271→  - 비활성화/활성화 처리 중 동일 매체에 대한 편성표 수정 요청은 대기 (낙관적 잠금)
   272→- **매체 비활성화 시 캠페인 상태 불일치**: 비활성화 시 해당 매체를 대상으로 하는 "집행중" 캠페인에 대해 알림을 발송한다
   273→  - 대상 매체가 **모두** 비활성화되면: 캠페인 담당자에게 "모든 대상 매체 비활성 — 실제 송출 불가" 경고 알림 발송
   274→  - 대상 매체 중 **일부만** 비활성화되면: "N개 매체 비활성 — 해당 매체 송출 중단" 정보 알림 발송
   275→  - 캠페인 상태 자체는 "집행중" 유지 (편성표 도메인에서 일시정지 처리)
   276→- **운영대행사가 권한 범위 밖 매체 전환 시도**: 버튼이 표시되지 않음
   277→
   278→#### 결과
   279→
   280→- 매체 상태 변경: "활성" ↔ "비활성"
   281→- 비활성 시: 해당 매체의 `적용중` 편성표를 `일시정지`로 전환, 광고 송출 중지. **비활성화 주체 ≠ 편성표 생성자인 경우** 편성표 생성자에게 "편성 N건이 일시정지됩니다" 알림 발송
   282→- 활성화 시: `일시정지` 편성표 중 종료일 미경과 → `적용중` 자동 복귀. 종료일 경과 → `종료` 처리

   123→6. 이후 재생 시 해당 구좌는 네이버 Ad Server가 실시간으로 광고를 결정한다
   124→
   125→#### 이런 경우에는
   126→
   127→- **매체에 네이버 SSP 연동이 안 된 상태**: "네이버 Ad Server 연동" 토글 비활성화. "먼저 매체의 SSP 연동을 설정해주세요" 툴팁
   128→- **모든 구좌를 네이버 Ad Server 연동으로 설정**: 허용. 전체 프로그래매틱 운영 가능
   129→- **네이버 Ad Server 연동 구좌를 다시 자체 소재로 전환**: 토글 해제 → 소재 선택 필요
   130→
   131→#### 결과
   132→
   133→- 해당 구좌가 네이버 Ad Server 연동 모드로 전환
   134→- 나머지 구좌는 기존 순환 재생 방식 유지

   138→### UC-24: 편성 충돌 감지 및 해소

   140→**누가**: 어드민, 매체사 또는 운영대행사
   141→**언제**: 편성표 생성/수정 시 동일 매체에 기간이 겹치는 편성이 감지될 때
   142→**전제조건**: UC-22에서 편성표 저장 시 충돌 감지

   144→#### 시나리오
   145→

   151→   - **A)** 현재 편성표의 우선순위를 올려서 기존 편성표를 대체
   152→   - **B)** 기존 편성표의 기간을 조정
   153→   - **C)** 현재 편성표 저장을 취소
   154→4. 사용자가 옵션을 선택하여 충돌을 해소한다

   168→### UC-25: 편성표 적용/종료
   169→
   170→**누가**: 시스템 (자동) 또는 매체사, 운영대행사 (수동 종료)
   171→**언제**: 편성표 시작일이 도래하거나, 종료일이 경과하거나, 사용자가 수동 종료할 때
   172→**전제조건**: UC-22로 편성표 생성 완료 ("예약됨" 상태)
   173→
   174→#### 시나리오 A — 자동 적용
   175→
   176→1. 시스템이 시작일이 도래한 편성표를 감지한다
   177→2. 편성표 상태를 "예약됨" → "적용중"으로 변경한다
   178→3. 해당 매체의 플레이어에 재생목록을 전송한다
   179→4. 플레이어가 재생목록의 구좌를 순서대로 순환 재생한다
   180→5. 편성 관리자에게 "편성 시작" 알림 발송
   181→
   182→#### 시나리오 B — 자동 종료
   183→
   184→1. 시스템이 종료일이 경과한 편성표를 감지한다
   185→2. 편성표 상태를 "적용중" → "종료"로 변경한다
   186→3. 편성 관리자에게 "편성 종료" 알림 발송
   187→
   188→#### 시나리오 C — 수동 종료
   189→
   190→1. 매체사 또는 운영대행사가 편성 관리 > 편성표 상세에서 "종료" 버튼을 클릭한다
   191→2. 시스템이 확인 모달: "이 편성표를 즉시 종료하시겠습니까?"
   192→3. 확인 시 편성표 상태 "적용중" → "종료"
   193→
   194→#### 이런 경우에는
   195→
   196→- **적용 시작 시 플레이어 오프라인**: 편성표는 "적용중"으로 전환되지만 실제 재생 불가. 매체 오프라인 알림 별도 발송 (UC-11 연계)
   197→- **적용 시작일 전 대상 매체가 비활성화됨**: 편성표는 '예약됨' 상태를 유지한다. 편성 담당자에게 '대상 매체 비활성 — 시작일에 적용되지 않을 수 있습니다' 알림 발송. 시작일 도래 시 매체가 여전히 비활성이면 '예약됨' → '적용중' → '일시정지'로 전환한다
   198→- **적용중 편성표의 재생목록을 수정**: 수정 후 저장 시 플레이어에 업데이트된 재생목록 즉시 전송
   199→- **종료 후 재적용**: 불가. 새 편성표 생성 필요
   200→- **편성표 vs 플레이어 상태 불일치**: CMS 편성표 상태는 "의도(intent)"이며, 플레이어 보고 상태는 "실제(actual)"이다. 플레이어는 현재 재생 중인 편성 버전을 주기적으로 CMS에 보고한다 (heartbeat). 의도와 실제가 불일치하면 편성표 상세에 "동기화 미완료" 배지를 표시한다. 불일치 지속 시 편성 담당자에게 알림 발송
   201→- **동시 편성표 수정 (낙관적 잠금)**: UC-22와 동일하게 버전 기반 낙관적 잠금 적용. 버전 불일치 시 "다른 사용자가 이 편성표를 수정했습니다. 최신 내용을 확인 후 다시 저장해주세요" 안내
   202→- **매체에 현재 적용 중인 편성표가 없음**: 플레이어는 "대기 화면"을 표시한다. 대기 화면은 매체사별로 설정 가능 (기본값: 시스템 기본 대기 이미지). CMS 매체 상세에서 "현재 편성 없음 — 대기 화면 송출 중" 상태를 표시한다
   203→- **편성 관련 Slack 알림 발송 실패**: 발송 실패 시 해당 실패 사실을 인앱 알림으로 발송한다. 인앱 알림 내용: "Slack 알림 전송 실패 — 인앱에서 확인해주세요: [원본 알림 내용 요약]"
   204→- **플레이어 전송 실패 재시도**: 플레이어가 온라인 상태인데 재생목록 전송이 실패하면, 지수 백오프로 최대 3회 재시도한다 (1분, 5분, 15분). 3회 모두 실패 시: 편성표 상세에 "동기화 미완료" 배지를 표시하고, 편성 담당자에게 알림 발송. 플레이어가 다음 heartbeat에서 버전 불일치를 보고하면 CMS가 재전송을 시도한다
   205→
   206→#### 결과
   207→
   208→- 적용: "예약됨" → "적용중". 매체에서 재생 시작
   209→- 종료: "적용중" → "종료". 매체에서 재생 중단

   213→## 5. 도메인 규칙
   214→
   215→### 5.1 편성표 상태 FSM
   216→
   217→```
   218→[예약됨] ──(시작일 도래)──→ [적용중] ──(종료일 경과 또는 수동 종료)──→ [종료]
   219→                               │        ↑
   220→                          (매체 비활성화) (매체 활성화 + 종료일 미경과)
   221→                               ↓        │
   222→                           [일시정지] ───┘
   223→                               │
   224→                          (종료일 경과)──→ [종료]
   225→```
   226→
   227→**상태**:
   228→| 상태 | 설명 | 허용 동작 |
   229→|------|------|----------|
   230→| 예약됨 | 편성표가 생성되어 저장된 상태. 적용 시작일 도래 전 | 수정, 삭제 |
   231→| 적용중 | 매체에서 재생 중 | 재생목록 수정, 수동 종료 |
   232→| 일시정지 | 매체 비활성화로 송출 중단, 매체 활성화 시 자동 재개 가능 | 조회, 매체 활성화 대기 |
   233→| 종료 | 재생 완료/중단 | 조회만 |
   234→
   235→**전이**:
   236→| From → To | 트리거 | 가드 | 효과 |
   237→|-----------|--------|------|------|
   238→| 예약됨 → 적용중 | 시작일 도래 | 대상 매체 활성 상태 | 플레이어에 재생목록 전송, 알림 |
   239→| 적용중 → 종료 | 종료일 경과 (자동) | — | 재생 중단, 알림 |
   240→| 적용중 → 종료 | 사용자가 수동 종료 | 권한 보유 | 즉시 재생 중단, 알림 |
   241→| 적용중 → 일시정지 | 대상 매체 비활성화 | — | 송출 중지. 비활성화 주체 ≠ 편성 생성자이면 생성자에게 알림 (UC-12 참조) |
   242→| 일시정지 → 적용중 | 대상 매체 활성화 | 종료일 미경과 | 플레이어에 재생목록 재전송, 송출 자동 재개 |
   243→| 일시정지 → 종료 | 종료일 경과 (자동) | — | 종료 처리. 매체 활성화 시에도 복구 불가 |
   244→
   245→> 매체 상태(활성/비활성)와 편성표 상태의 연계는 매체 관리 UC-12를 참조한다.
   246→> 매체 비활성화 시 "적용중" → "일시정지", 매체 활성화 시 종료일 재검사 후 복귀.
   247→
   248→**불변식**:
   249→- INV-18: 편성표는 최소 1개의 재생목록을 가져야 함
   250→- INV-19: 편성표는 최소 1개의 대상 매체를 가져야 함
   251→- INV-20: 비활성 매체에 편성표 적용 불가
   252→- INV-21: 재생목록의 구좌는 1개 이상이어야 함 (빈 재생목록 불가)

   254→### 5.2 구좌 상태
   255→
   256→| 상태 | 설명 | 재생 시 동작 | UI 표시 |
   257→|------|------|------------|---------|
   258→| 정상 | 소재가 배정된 구좌 | 소재 재생 | 기본 표시 |
   259→| 삭제됨 | 배정 소재가 삭제된 구좌 | 스킵 (구좌 구조 보존) | 빨간 점선 + "소재 삭제됨 — 교체 필요" |
   260→| 교체중 | 배정 소재의 파일 교체 후 재검수 진행 중 | 기존 파일로 재생 유지 | "교체중" 라벨 |
   261→| 네이버 Ad Server 연동 | 네이버 Ad Server가 소재를 결정하는 구좌 | Ad Server 실시간 결정 | "네이버 Ad Server" 라벨 |

   265→### 5.3 재생 모델
   266→
   267→- **기본**: 순환 재생 — 재생목록의 구좌를 순서대로 무한 반복
   268→- **확장**: 네이버 Ad Server 연동 구좌는 Ad Server가 실시간 결정
   269→- 나머지 구좌는 기존 순환 재생 유지
   270→- 별도 LoopPolicy 엔티티 없음

   273→
   274→- SCH-BR-01: 구좌의 재생 시간 = 배정된 소재의 재생 시간 (구좌 자체 시간 설정 없음)
   275→- SCH-BR-02: 동일 매체에 복수 편성표 적용 가능. 우선순위로 결정
   276→- SCH-BR-03: 편성 충돌 = 동일 매체 + 겹치는 기간 + 우선순위 미결정

   278→- SCH-BR-05: 편성표 상태 자동 전환은 시작일시·종료일시를 분 단위로 지정하며, 해당 시각에 즉시 전환한다

   282→## 6. 미결 사항
   283→
   284→- 편성표 우선순위의 상세 동작 미정의 (우선순위 높은 편성표만 재생? 혼합 재생?)
   285→- 재생목록 수정 시 적용중 편성표에 즉시 반영되는 메커니즘 상세 미정의
   286→- 종료된 편성표 이후 매체에 표시할 기본 콘텐츠(대기 화면) 미정의
   287→- 편성표 없는 매체의 기본 동작 미정의 (블랙스크린? 기본 이미지?)
   288→- 재생목록 삭제 시 해당 재생목록을 참조하는 편성표 처리 미정의
   289→- 네이버 Ad Server 응답 지연/실패 시 해당 구좌의 fallback 동작 미정의
   290→- 플레이어에 재생목록 전송 실패 시 재시도 정책 미정의
   291→- 네이버 Ad Server 연동 구좌와 자체 구좌 혼합 시 전체 순환 재생 시간 계산 방식 미정의 (네이버 구좌 시간은 가변)
   292→- **[인터뷰 도출]** UC-24 편성 충돌 감지 시점 — 현재 저장 후 감지이나, 인터뷰에서 매체사 팀장이 "저장하고 나서야 충돌을 알게 된다"는 패턴 확인됨. 저장 전 실시간 충돌 경고(입력 중 감지) 방식의 구현 타당성 검토 필요
   293→- **[인터뷰 도출]** 편성 타임라인 시각화 미정의 — 매체사 팀장이 CMS 편성 화면만으로 전체 일정 파악이 안 되어 엑셀 병행 관리 중. 매체별 편성표를 간트 차트/타임라인 뷰로 표시하는 기능 요구 확인됨
   294→- **[인터뷰 도출]** 재생목록-편성 2단계 구조의 온보딩 가이드 미정의 — 운영대행사 현장 매니저가 재생목록을 먼저 만들어야 편성이 가능하다는 구조를 파악하지 못해 편성 누락 발생. 첫 편성 생성 시 단계 안내 또는 인컨텍스트 도움말 필요
   295→- "삭제됨" 구좌 스킵 시 플레이어 에러 핸들링 상세 — 플레이어 설계 단계에서 정의
