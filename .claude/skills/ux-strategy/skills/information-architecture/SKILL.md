# Information Architecture

## Context

$ARGUMENTS — 대상 프로젝트 또는 집중할 도메인. 생략 시 전체 시나리오 기반으로 진행.

전략(strategize)과 개별 화면 설계(design-screen) 사이의 구조적 갭을 메운다. 역할별 접근 범위가 다른 시스템(예: CMS, 어드민, SaaS)에서 이 단계를 건너뛰면 화면 간 관계와 네비게이션이 파편화된다.

## Domain Context

- **화면 인벤토리**는 시나리오 UC에서 도출한다. UC당 최소 1개 화면이 존재한다
- **역할별 접근 매트릭스**는 페르소나에서 도출한다. 동일 화면이라도 역할에 따라 보이는 범위가 다를 수 있다
- **네비게이션 구조**는 GNB(Global Navigation Bar) / LNB(Local Navigation Bar) 2계층이 기본이다. 3계층 이상은 인지 부하를 높인다
- **플로우 다이어그램**은 주요 태스크 단위로 작성한다. 모든 화면을 연결하지 않는다
- **opportunity-framework** 결과가 있으면 설계 우선순위 결정에 활용한다

## Instructions

### Step 1: 입력 수집

아래 문서를 순서대로 읽는다. 없는 문서는 건너뛴다:

1. `docs/scenarios/` — 시나리오 파일 전체 (UC 목록 추출)
2. `docs/research/personas/` — 페르소나 파일 (역할 목록 추출)
3. 전략 문서 (opportunity-framework 결과가 있으면 우선순위 참조)

### Step 2: 화면 인벤토리

시나리오 UC에서 화면 목록을 추출한다:

```
| # | 화면명 | 출처 UC | 핵심 기능 | 비고 |
|---|--------|---------|-----------|------|
| 1 | 대시보드 | UC-01, UC-03 | 현황 요약, 빠른 접근 | 랜딩 화면 |
| 2 | ... | ... | ... | ... |
```

규칙:
- UC당 최소 1개 화면. 하나의 UC가 여러 화면에 걸칠 수 있다
- 화면명은 사용자가 이해할 수 있는 한국어 명칭 사용
- CRUD 단위로 분리하지 않는다 (목록+상세는 1화면 또는 2화면 — 복잡도에 따라 판단)

### Step 3: 역할별 접근 매트릭스

페르소나 × 화면 매트릭스를 작성한다:

```
| 화면 | 역할A | 역할B | 역할C | 역할D |
|------|:-----:|:-----:|:-----:|:-----:|
| 대시보드 | ✅ 전체 | ✅ 본인 | ❌ | ✅ 전체 |
| 콘텐츠 관리 | ✅ CRUD | ✅ R | ❌ | ✅ CRUD |
```

- ✅ = 접근 가능 (범위 명시)
- ❌ = 접근 불가
- 🔒 = 조건부 접근 (조건 명시)

### Step 4: 네비게이션 구조

GNB/LNB 계층을 설계한다:

```
GNB
├── 대시보드
├── 콘텐츠
│   ├── [LNB] 캠페인 관리
│   ├── [LNB] 미디어 라이브러리
│   └── [LNB] 스케줄
├── 디바이스
│   ├── [LNB] 디바이스 목록
│   └── [LNB] 디바이스 그룹
└── 설정
    ├── [LNB] 계정
    └── [LNB] 시스템
```

규칙:
- GNB 항목은 7±2개 이내
- LNB는 GNB 선택 시 노출. 10개 이내
- 3계층 이상 피한다. 불가피하면 이유를 명시

### Step 5: 화면 간 플로우

주요 태스크(3~5개)에 대한 플로우를 작성한다:

```
### 태스크: [태스크명]
[시작 화면] → [중간 화면] → [완료 화면]
                  ↓ (조건)
              [분기 화면]
```

- 모든 화면을 연결하지 않는다. 핵심 태스크 플로우만
- 분기 조건이 있으면 명시
- 에러/예외 경로는 포함하지 않는다 (개별 화면 설계에서 다룸)

### Step 6: 우선순위 매핑

opportunity-framework 결과가 있으면 연계하여 설계 순서를 결정한다:

```
| 우선순위 | 화면 | 이유 |
|:--------:|------|------|
| 1 | ... | opportunity score 최고 + 핵심 플로우 |
| 2 | ... | ... |
```

없으면 사용자에게 우선순위 기준을 질문한다.

### Step 7: 산출물 저장

결과를 `docs/design/information-architecture.md`에 저장한다.

저장 후 안내:

```
IA 문서가 docs/design/information-architecture.md에 저장되었습니다.

다음 단계:
- 개별 화면 설계: `/design-screen [화면명]`
- 아이디어 발산이 필요하면: `/brainstorming [화면명]`
```

## Common Mistakes

| 실수 | 결과 | 수정 |
|------|------|------|
| 시나리오 없이 화면 목록 작성 | 근거 없는 구조 | 반드시 UC에서 도출 |
| 역할 구분 없이 단일 네비게이션 | 권한 문제 후발 | 접근 매트릭스 먼저 |
| 모든 화면을 플로우에 연결 | 읽기 어려운 다이어그램 | 핵심 태스크만 |
| CRUD 단위 화면 분리 | 화면 수 폭발 | 복잡도 기반 판단 |
