# 사용자 관리 시나리오

## 1. 개요

CMS에 접근하는 모든 사용자의 계정 생성, 권한 부여, 비활성화를 관리한다. 이메일 초대 기반 가입 방식을 사용하며, 자체 회원가입은 지원하지 않는다.

## 4. 상세 시나리오

### UC-01: 어드민이 매체사 담당자를 초대

**누가**: 어드민
**언제**: 새로운 매체사를 CMS에 온보딩할 때
**전제조건**: 어드민이 로그인된 상태. 대상 매체사가 UC-06(매체사 등록)으로 등록 완료되었거나, 초대와 동시에 매체사 등록을 진행

#### 시나리오

1. 어드민이 사용자 관리 > 초대 버튼을 클릭한다
2. 초대 대상의 이메일 주소를 입력한다
3. 역할을 "매체사"로 지정한다
4. 소속 매체사를 선택한다
5. 초대장 발송 버튼을 클릭한다
6. 시스템이 초대 메일을 발송하고, 초대 상태를 "발송됨"으로 기록한다
7. 초대 목록에 해당 건이 표시된다

#### 이런 경우에는

- **이미 가입된 이메일 (활성 상태)**: "이미 등록된 사용자입니다" 안내. 초대 발송 차단
- **이미 가입된 이메일 (비활성 상태)**: "비활성화된 사용자입니다. 활성화하시겠습니까?" 확인 → 활성화 처리 (새 초대 불필요)
- **동일 이메일로 이미 초대 발송됨 (미가입 상태)**: "이미 초대가 발송된 이메일입니다. 재발송하시겠습니까?" 확인 후 재발송 가능
- **이메일 형식 오류**: 필드 하이라이트 + "올바른 이메일 형식을 입력해주세요" 안내
- **소속 매체사 미선택**: 다음 단계 버튼 비활성화

#### 결과

- 초대 레코드 생성: 상태 "발송됨", 유효기간 7일
- 대상 이메일로 가입 링크 포함 초대 메일 발송
- 인앱 알림: 초대 주체(어드민)에게 "초대 발송 완료" 알림

---

### UC-02: 초대받은 사용자가 가입 완료

**누가**: 초대받은 사용자
**언제**: 초대 메일의 가입 링크를 클릭했을 때
**전제조건**: UC-01 또는 UC-03에서 초대가 발송된 상태. 초대 유효기간 내

#### 시나리오

1. 사용자가 초대 메일의 가입 링크를 클릭한다
2. 가입 페이지가 열린다. 이메일은 초대 시 지정된 값으로 고정 표시
3. 이름, 비밀번호를 입력한다
4. 이용약관에 동의한다
5. 가입 완료 버튼을 클릭한다
6. 시스템이 계정을 생성하고, 초대 시 지정된 역할과 소속을 자동 부여한다
7. CMS 메인 화면으로 이동한다

#### 이런 경우에는

- **초대 유효기간 만료**: "초대가 만료되었습니다. 관리자에게 재발송을 요청해주세요" 안내. 가입 불가
- **비밀번호 정책 미충족**: 필드 하이라이트 + 정책 안내 (최소 8자, 영문+숫자+특수문자)
- **이미 가입 완료된 초대 링크 재클릭**: "이미 가입이 완료되었습니다" 안내 → 로그인 페이지로 이동

#### 결과

- 사용자 계정 생성: 역할(매체사/운영대행사/영업대행사), 소속 매체사 자동 설정
- 초대 레코드 상태: "발송됨" → "가입완료"
- 인앱 알림: 초대 주체에게 "xxx님이 가입을 완료했습니다" 알림

---

### UC-03: 매체사가 운영대행사/영업대행사를 추가 초대

**누가**: 매체사
**언제**: 소속 매체의 운영 또는 영업을 위임할 외부 담당자를 초대할 때
**전제조건**: 매체사 사용자가 로그인된 상태. UC-02를 통해 가입 완료된 매체사 계정

#### 시나리오

1. 매체사가 사용자 관리 > 초대 버튼을 클릭한다
2. 초대 대상의 이메일 주소를 입력한다
3. 역할을 선택한다: "운영대행사" 또는 "영업대행사"
4. 권한 범위를 설정한다:
   - 운영대행사: 접근 가능한 매체 또는 매체 그룹을 선택
   - 영업대행사: 소재 도메인 접근 범위를 설정 (전체 또는 본인 소속 데이터만)
5. 초대장 발송 버튼을 클릭한다
6. 시스템이 초대 메일을 발송한다

#### 이런 경우에는

- **매체사에 등록된 매체가 없는 상태에서 운영대행사 초대**: "먼저 매체를 등록해주세요" 안내. 초대 발송 차단
- **이미 가입된 이메일**: UC-01과 동일 처리
- **동일 사용자를 다른 역할로 재초대**: "이미 xxx 역할로 등록된 사용자입니다. 역할을 변경하시겠습니까?" 확인 → UC-04로 이동

#### 결과

- 초대 레코드 생성: 역할, 권한 범위 포함
- 초대 메일 발송
- 인앱 알림: 초대 주체(매체사)에게 발송 완료 알림

### UC-04: 사용자 권한 변경

**누가**: 어드민 또는 매체사
**언제**: 기존 사용자의 역할 또는 접근 범위를 변경해야 할 때
**전제조건**: 대상 사용자가 가입 완료(활성) 상태. 변경 주체가 대상 사용자에 대한 관리 권한 보유

#### 시나리오

1. 어드민 또는 매체사가 사용자 관리 > 사용자 목록에서 대상 사용자를 선택한다
2. 상세 정보 화면에서 "권한 변경" 버튼을 클릭한다
3. 변경할 항목을 수정한다:
   - 역할 변경 (운영대행사 ↔ 영업대행사)
   - 접근 가능 매체/매체 그룹 범위 변경
4. 변경 사유를 입력한다 (선택)
5. 저장 버튼을 클릭한다
6. 시스템이 권한을 즉시 반영한다

#### 이런 경우에는

- **매체사가 어드민 권한 사용자를 변경 시도**: 변경 버튼이 표시되지 않음 (권한 부족)
- **매체사가 타 매체사 소속 사용자를 변경 시도**: 사용자 목록에 표시되지 않음
- **운영대행사→영업대행사 변경 시 기존 편성 관련 데이터 존재**: "기존 편성 관리 권한이 회수됩니다. 계속하시겠습니까?" 확인

#### 결과

- 사용자 권한 즉시 변경 반영
- 변경 이력 기록 (변경 주체, 변경 전/후 값, 변경 일시)
- 대상 사용자에게 "권한이 변경되었습니다" 인앱 알림

### UC-05: 사용자 비활성화/탈퇴

**누가**: 어드민 또는 매체사
**언제**: 사용자의 CMS 접근을 중지하거나, 사용자가 탈퇴를 요청할 때
**전제조건**: 대상 사용자가 활성 상태. 비활성화 주체가 관리 권한 보유

#### 시나리오

1. 어드민 또는 매체사가 사용자 관리 > 사용자 목록에서 대상 사용자를 선택한다
2. "비활성화" 버튼을 클릭한다
3. 시스템이 확인 모달을 표시한다: "이 사용자를 비활성화하시겠습니까?"
4. 확인 버튼을 누르면 사용자 상태가 "활성" → "비활성"으로 변경된다
5. 비활성화된 사용자는 즉시 로그아웃 처리되며, 이후 로그인 불가

#### 이런 경우에는

- **매체사의 유일한 활성 사용자를 비활성화 시도**: "이 매체사의 마지막 활성 사용자입니다. 비활성화하면 해당 매체사를 관리할 수 없게 됩니다. 계속하시겠습니까?" 경고
- **비활성화된 사용자를 다시 활성화**: 사용자 상세에서 "활성화" 버튼 → 즉시 활성화 → 기존 역할/권한 복원
- **사용자가 본인 탈퇴 요청**: 설정 > 계정 > 탈퇴 요청 → 어드민 또는 매체사에게 탈퇴 요청 알림 발송 → 어드민 또는 매체사가 승인 시 비활성화 처리

#### 결과

- 사용자 상태: "활성" → "비활성"
- 즉시 세션 무효화 (로그아웃 처리)
- 비활성화 이력 기록 (주체, 일시, 사유)
- 비활성화된 사용자의 데이터(등록한 소재, 캠페인 등)는 삭제되지 않고 유지

## 5. 도메인 규칙

### 5.1 사용자 상태 FSM

```
[초대발송됨] ──(가입완료)──→ [활성]
     │                        │    ↑
     │(만료)                (비활성화) (활성화)
     ↓                        ↓    │
[초대만료]                  [비활성] ─┘
```

**상태**:
| 상태 | 설명 | 허용 동작 |
|------|------|----------|
| 초대발송됨 | 초대 메일 발송, 미가입 | 재발송, 만료 |
| 초대만료 | 유효기간 경과 | 재발송(새 초대 생성) |
| 활성 | 가입 완료, CMS 사용 가능 | 로그인, 기능 사용, 비활성화 |
| 비활성 | CMS 접근 차단 | 활성화(관리자) |

**전이**:
| From → To | 트리거 | 가드 | 효과 |
|-----------|--------|------|------|
| 초대발송됨 → 활성 | 사용자가 가입 완료 | 유효기간 내, 필수정보 입력 | 계정 생성, 역할/소속 부여 |
| 초대발송됨 → 초대만료 | 유효기간 경과 (7일) | — | 가입 링크 무효화 |
| 초대만료 → 초대발송됨 | 어드민 또는 매체사가 재발송 | — | 새 초대 레코드 생성 |
| 활성 → 비활성 | 어드민 또는 매체사가 비활성화 | 관리 권한 보유 | 세션 무효화, 로그인 차단 |
| 비활성 → 활성 | 어드민 또는 매체사가 활성화 | 관리 권한 보유 | 기존 역할/권한 복원 |

**불변식**:
- INV-01: 모든 활성 사용자는 정확히 하나의 역할(어드민/매체사/운영대행사/영업대행사)을 가진다
- INV-02: 매체사/운영대행사/영업대행사는 반드시 하나의 소속 매체사에 연결된다
- INV-03: 영업대행사는 소재 도메인 외 기능에 접근할 수 없다
- INV-04: 초대 유효기간은 발송 시점으로부터 7일이다

### 5.2 비즈니스 규칙

- USR-BR-01: 자체 회원가입 불가. 반드시 초대를 통해서만 계정 생성
- USR-BR-02: 어드민은 시스템에 직접 생성 (초대 프로세스 외)
- USR-BR-03: 한 이메일 = 한 계정. 동일 이메일로 복수 계정 불가
- USR-BR-04: 비활성화된 사용자의 기존 데이터(소재, 캠페인 등)는 삭제되지 않음
- USR-BR-05: 권한 변경은 즉시 반영. 다음 요청부터 새 권한 적용

## 6. 미결 사항

- 초대 유효기간 7일이 적정한지 확인 필요. 만료 기준: 발송 시점 +7일 00:00:00 (자정 기준? 정확히 168시간?)
- 어드민 계정의 초기 생성 방법 (시드 데이터? 별도 관리 콘솔?) 미정의
- 비밀번호 정책 상세 (최소 길이, 복잡도, 변경 주기) 미정의
- 한 사용자가 복수 매체사에 소속 가능한지 미정의 (현재 설계: 1인 1매체사)
- 탈퇴 시 개인정보 처리 방안 (보존 기간, 익명화) 미정의
- 운영대행사→영업대행사 역할 전환 시 기존 편성 작업물의 소유자 처리 방안 미정의
- 매체사의 유일한 활성 사용자 비활성화 후 복구 주체: 어드민만 가능한지 확인 필요
