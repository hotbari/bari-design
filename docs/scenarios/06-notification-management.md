# 알림 관리 시나리오

## 1. 개요

CMS 내에서 발생하는 주요 이벤트를 사용자에게 실시간으로 전달한다. 3개 채널(인앱, 브라우저 푸시, Slack)을 지원하며, 사용자별로 채널·카테고리 조합을 설정할 수 있다.

## 4. 상세 시나리오

### UC-26: 인앱 알림 수신 및 읽음 처리

**누가**: 모든 페르소나 (어드민, 매체사, 운영대행사, 영업대행사)
**언제**: CMS 사용 중 이벤트가 발생하거나, 알림 센터를 확인할 때
**전제조건**: 로그인 상태

#### 시나리오

1. CMS 이벤트 발생 시 시스템이 이벤트 카탈로그(§5.1)에 정의된 수신 대상에게 인앱 알림을 생성한다
2. 헤더의 bell 아이콘에 읽지 않은 알림 수 배지가 표시된다
3. 사용자가 bell 아이콘을 클릭하면 알림 센터가 열린다
4. 알림 목록이 최신순으로 표시된다:
   - 읽지 않은 알림: 강조 표시
   - 읽은 알림: 일반 표시
5. 알림을 클릭하면:
   - 해당 알림이 "읽음" 처리된다
   - 관련 화면으로 이동한다 (예: 캠페인 상세, 소재 상세 등)
6. "전체 읽음" 버튼으로 모든 알림을 일괄 읽음 처리할 수 있다

#### 이런 경우에는

- **알림 수신 대상이 오프라인(비로그인)**: 알림 레코드는 생성됨. 다음 로그인 시 확인 가능
- **90일 경과한 알림**: 자동 삭제 (보존 기간: 90일)
- **영업대행사의 알림**: 소재 관련 이벤트(검수 완료, 검수 실패)만 수신
- **대량 알림 발생 (매체 100대 동시 오프라인 등)**: 동일 카테고리 + 동일 매체사 범위 내, 5분 시간 윈도우로 그룹핑하여 "N건의 매체 오프라인" 형태로 표시. 그룹 클릭 시 개별 매체 목록 펼침

#### 이벤트 카탈로그

| 카테고리 | 이벤트 | 기본 채널 | 수신 대상 |
|---------|--------|----------|----------|
| 소재 검수 | 검수 완료 | 인앱 | 소재 등록자 (UC-13 참조) |
| 소재 검수 | 검수 실패 | 인앱, 브라우저 푸시 | 소재 등록자 (UC-13 참조) |
| 소재 검수 | 수동검수대기 | 인앱 | 검수 권한자 ("사용자 확인 필요"), 소재 등록자 ("소재가 수동 검수 대기 중입니다. 검수 담당자가 확인 후 처리 예정입니다") |
| 매체 상태 | 매체 오프라인 | 인앱, 브라우저 푸시, Slack | 해당 매체의 매체사 + 운영대행사 |
| 매체 상태 | 매체 이상 감지 | 인앱, 브라우저 푸시, Slack | 해당 매체의 매체사 + 운영대행사 |
| 매체 상태 | 매체 복구 | 인앱 | 해당 매체의 매체사 + 운영대행사 |
| 편성 | 편성 충돌 | 인앱, 브라우저 푸시 | 편성표 생성자 (UC-22 참조). 생성자 권한 밖 충돌 시 매체사도 추가 |
| 편성 | 편성 시작 | 인앱 | 편성표 생성자 (UC-22 참조) |
| 편성 | 편성 종료 | 인앱 | 편성표 생성자 (UC-22 참조) |
| 편성 | 편성 일시정지 (매체 비활성화) | 인앱, 브라우저 푸시 | 편성표 생성자 (UC-22 참조) (비활성화 주체 ≠ 편성표 생성자 (UC-12 참조)인 경우에만) |
| 캠페인 | 캠페인 시작일 매체 비활성 | 인앱 | 캠페인 등록자 |
| 캠페인 | 캠페인 완료 — 연결 편성표 적용중 | 인앱 | 캠페인 등록자 |
| 사용자 | 초대 발송 | 인앱 | 초대 주체 |
| 사용자 | 초대 만료 임박 (24h 전) | 인앱 | 초대 주체 |
| 사용자 | 가입 완료 | 인앱 | 초대 주체 |
| 리포트 | 리포트 발송 완료 | 인앱 | 리포트 설정자 |
| 리포트 | 리포트 발송 실패 | 인앱, 브라우저 푸시 | 리포트 설정자 |
| 소재 | 소재 운영 기간 만료 임박 (3일 전) | 인앱 | 소재 등록자, 해당 소재가 편성된 편성표 생성자 |

> 캠페인 승인/반려 이벤트 제거됨 (확정 의사결정: 승인 워크플로우 없음)

#### 결과

- 알림 레코드 생성: 읽지 않음 상태, 90일 보존
- bell 아이콘 배지 업데이트

### UC-27: 브라우저 푸시 권한 설정

**누가**: 모든 페르소나
**언제**: 브라우저 푸시 알림을 받거나 끄고 싶을 때
**전제조건**: 로그인 상태. 브라우저가 Web Push API를 지원

#### 시나리오

1. 사용자가 설정 > 알림 설정 메뉴를 클릭한다
2. 알림 채널 설정이 표시된다:
   - 인앱 알림: 항상 활성 (비활성화 불가)
   - 브라우저 푸시: 토글 (기본 OFF)
3. 브라우저 푸시 토글을 ON으로 변경한다
4. 브라우저가 알림 권한 요청 팝업을 표시한다
5. "허용"을 선택하면 브라우저 푸시가 활성화된다
6. 카테고리별로 브라우저 푸시 수신 여부를 설정한다 (기본: 긴급 우선순위 이벤트만 — §5.1 이벤트 카탈로그의 '긴급' 우선순위 항목)

#### 이런 경우에는

- **브라우저에서 "차단" 선택**: 브라우저 푸시 활성화 불가. "브라우저 설정에서 알림을 허용해주세요" 안내
- **브라우저가 Web Push API 미지원**: 브라우저 푸시 설정 항목 비표시. "현재 브라우저는 푸시 알림을 지원하지 않습니다" 안내
- **기존에 허용했다가 토글 OFF**: 구독 해제. 이후 푸시 미발송

#### 결과

- 브라우저 푸시 구독 등록/해제
- 카테고리별 수신 설정 저장

### UC-28: Slack Webhook 설정

**누가**: 어드민
**언제**: 운영팀 Slack 채널에 CMS 알림을 연동할 때
**전제조건**: 어드민 로그인 상태. Slack Webhook URL 보유

#### 시나리오

1. 어드민이 설정 > Slack 연동 메뉴를 클릭한다
2. Slack Webhook URL을 입력한다
3. 테스트 발송 버튼을 클릭한다
4. 시스템이 Slack에 테스트 메시지를 발송한다
5. 발송 성공 시 "연동 완료" 표시
6. 카테고리-채널 매핑을 설정한다:
   - 매체 상태 이벤트 → #ops-alerts 채널
   - 편성 충돌 이벤트 → #scheduling 채널
   - (카테고리별로 다른 Webhook URL/채널 매핑 가능)
7. 저장 버튼을 클릭한다

#### 이런 경우에는

- **잘못된 Webhook URL**: 테스트 발송 실패. "Webhook URL이 올바르지 않습니다" 안내
- **Slack 서버 장애로 발송 실패**: "Slack 발송에 실패했습니다. 잠시 후 다시 시도해주세요" 안내. 알림은 인앱에는 정상 전달
- **복수 Webhook URL 등록**: 허용. 카테고리별로 다른 채널에 발송 가능
- **Slack 발송 실패 시 재시도**: 지수 백오프로 최대 3회 재시도 (1분, 5분, 15분)
- **3회 재시도 모두 실패**: 원본 알림 내용을 인앱 알림으로 발송. 어드민에게 "Slack 전송 실패" 별도 알림 발송. 해당 Webhook의 연동 상태를 "전송 실패"로 변경
- **24시간 내 동일 Webhook 실패율 10% 초과**: 어드민에게 "Slack 연동 점검 필요" 긴급 알림 발송

#### 결과

- Slack Webhook 설정 저장
- 이후 해당 카테고리 이벤트 발생 시 Slack에 자동 발송

## 5. 도메인 규칙

### 5.1 알림 우선순위

**우선순위 정의**
- 긴급: 즉시 대응 필요. 광고 송출에 직접 영향 (예: 매체 오프라인, 검수 실패)
- 일반: 업무 흐름에 필요한 정보 (예: 검수 완료, 편성 적용 시작)
- 정보: 참고용 (예: 리포트 생성 완료)

| 우선순위 | 이벤트 유형 | 기본 채널 |
|:--------:|-----------|----------|
| 긴급 | 매체 오프라인, 매체 이상 감지, 검수 실패, 편성 충돌, 편성 일시정지, 캠페인 시작일 매체 비활성, 리포트 발송 실패 | 인앱 + 브라우저 푸시 + Slack |
| 일반 | 검수 완료, 수동검수대기, 편성 시작/종료, 가입 완료, 리포트 발송 완료, 소재 운영 기간 만료 임박, 캠페인 완료 — 연결 편성표 적용중 | 인앱 |
| 정보 | 초대 발송, 초대 만료 임박, 매체 복구 | 인앱 |

### 5.2 비즈니스 규칙

- NTF-BR-01: 인앱 알림은 항상 활성. 사용자가 비활성화할 수 없음
- NTF-BR-02: 브라우저 푸시는 사용자별 opt-in. 기본 OFF
- NTF-BR-03: Slack은 어드민만 설정 가능. 카테고리별 채널 매핑
- NTF-BR-04: 알림 보존 기간: 90일. 이후 자동 삭제
- NTF-BR-05: 알림 수신 대상은 이벤트 발생 시점의 사용자 역할과 매체 관리 범위를 기준으로 결정
- NTF-BR-06: 동일 카테고리 + 동일 매체사 범위 내 5분 시간 윈도우 이내 발생한 알림은 그룹핑하여 "N건의 [이벤트명]" 형태로 표시. 그룹 클릭 시 개별 목록 펼침
- NTF-BR-07: Slack 발송 큐에 rate limit 적용 — Webhook당 초당 최대 1건 발송. Slack API 429 응답 시 `Retry-After` 헤더를 준수하여 대기. 큐 적체 시 동일 카테고리 이벤트를 NTF-BR-06 그룹핑 규칙으로 묶어 단일 메시지로 발송

## 6. 미결 사항

- ~~알림 그룹핑 기준 미정의~~ → 동일 카테고리 + 동일 매체사 범위 내, 5분 시간 윈도우로 그룹핑. "N건의 매체 오프라인" 형태로 표시. 그룹 클릭 시 개별 매체 목록 펼침
- 알림 발송 실패 시 재시도 정책 미정의 (브라우저 푸시, Slack)
- 사용자별 카테고리 수신 설정의 상세 UI 미정의
- 알림 센터의 페이지네이션/무한 스크롤 방식 미정의
- 이메일 알림 채널 추가 여부 미정의 (현재: 인앱, 브라우저 푸시, Slack만)
- Slack 외 메신저(네이버웍스 등) 연동 필요 여부 미정의 (RFP에서 네이버웍스 협업도구 언급). 본 시나리오에서 "네이버"는 연동 설정 맥락에서는 "네이버 SSP", 광고 송출 맥락에서는 "네이버 Ad Server"로 구분
- **[인터뷰 도출]** 알림 센터 우선순위 탭 분리 UI 미정의 — §5.1에 긴급/일반/정보 우선순위가 정의되어 있으나, 인터뷰에서 매체사 팀장이 "Slack 알림이 많아지면 무뎌지고 긴급/일반 구분이 안 된다"는 패턴 확인됨. 알림 센터 내 탭 또는 시각적 그룹핑으로 우선순위를 명시적으로 구분하는 방식 확정 필요
- **[인터뷰 도출]** 역할별 알림 타입 제한 상세 정의 필요 — 영업대행사는 소재 검수 이벤트만 수신(UC-26 명시)이나, 매체 오프라인 알림을 영업대행사가 수신해야 하는지(광고 집행 차질 인지 목적) 미결. 운영대행사의 알림 범위도 "부여받은 매체"로만 제한되는지 엣지케이스 정의 필요
- **[인터뷰 도출]** 매체 오프라인 알림 수신 대상에 영업대행사 추가 확정 권고 — 영업대행사 인터뷰에서 "광고주가 먼저 장애를 인지하고 연락 온" 실사례 확인. 해당 매체에 소재가 편성된 영업대행사도 매체 오프라인/복구 알림 수신 대상에 포함해야 광고주 선제 대응 가능
- **[인터뷰 도출]** 소재 운영 기간 만료 알림 2단계화 — 현재 3일 전 1회만 정의. 신규 소재 제작-수령-업로드-검수 리드타임(약 3~5일) 고려 시 7일 전 사전 알림 추가 필요. 이벤트 카탈로그에 "소재 운영 기간 만료 사전 안내(7일 전)" 항목 추가 검토
