# 편성 관리 시나리오

## 1. 개요

광고 소재를 재생목록으로 구성하고, 편성표를 통해 매체에 배정하여 송출한다. 기본은 순환 재생(루프 기반 반복 재생)이며, 특정 구좌를 네이버 Ad Server와 연동하여 프로그래매틱 동작도 가능하다.

**핵심 흐름**: 소재 선택 → 재생목록 구성 → 편성표 생성(매체 배정 + 기간) → 매체에서 반복 재생

**용어 정의**
- 편성 관리: 이 도메인의 이름. 편성표와 재생목록의 생성·적용·종료를 관리하는 업무 영역
- 편성표: 특정 매체에 특정 기간 동안 적용되는 재생 스케줄 엔티티
- 재생목록: 편성표에 연결되는 구좌(Slot)의 순서 목록 엔티티
- 플레이어: 매체의 디바이스에서 실행되는 재생 소프트웨어. CMS로부터 재생목록을 수신하여 콘텐츠를 송출한다

## 4. 상세 시나리오

### UC-21: 재생목록 생성

**누가**: 매체사 또는 운영대행사
**언제**: 매체에 송출할 소재들의 재생 순서를 구성할 때
**전제조건**: 로그인 상태. "검수완료" 상태의 소재가 1개 이상 존재 (UC-14 완료)

#### 시나리오

1. 매체사가 편성 관리 > 재생목록 > 새 목록 버튼을 클릭한다
2. 재생목록명을 입력한다 (필수)
3. 소재를 추가한다:
   - 소재 목록에서 "검수완료" 상태의 소재만 표시
   - 드래그&드롭으로 순서 배치
   - 각 소재가 하나의 구좌(Slot)를 차지
   - **더미(스킵) 구좌 배치 가능**: 소재 대신 "더미(스킵)"를 배치하면 재생 시 해당 구좌를 스킵
4. 구좌 순서를 조정한다 (드래그&드롭)
5. 저장 버튼을 클릭한다

> **분할 구좌**: 더미 구좌를 활용하여 노출 빈도를 조절한다.
> - 1구좌: 모든 슬롯에 소재 → 매 사이클 1회 노출
> - 반구좌: 1소재 + 1더미 → 2사이클에 1회 노출
> - 1/3구좌: 1소재 + 2더미 → 3사이클에 1회 노출
> - 1/4, 1/5 등 동일 패턴으로 확장

#### 이런 경우에는

- **검수완료 소재가 0건**: "사용 가능한 소재가 없습니다. 먼저 소재를 업로드하고 검수를 완료해주세요" 안내
- **동일 소재를 복수 구좌에 배치**: 허용. 같은 소재가 여러 번 반복 재생 가능
- **재생목록에 소재 0건인 상태로 저장 시도**: "최소 1개의 소재를 추가해주세요" 안내
- **더미만으로 구성된 재생목록 저장 시도**: "더미 구좌만으로는 재생목록을 구성할 수 없습니다. 최소 1개의 소재를 추가해주세요" 안내. 저장 차단
- **재생목록에 포함된 소재가 이후 삭제됨**: 해당 구좌가 "삭제됨" 상태로 전환. 빨간 점선 테두리 + "소재 삭제됨 — 교체 필요" 라벨 표시. "교체" 버튼으로 즉시 대체 가능. 재생 시 해당 구좌 스킵. 편성 관리자에게 알림 발송
- **편성표가 참조 중인 재생목록 삭제 시도**: 삭제를 차단한다. "이 재생목록을 사용 중인 편성표가 N건 있습니다. 편성표에서 먼저 교체해주세요" 안내. 참조 편성표 목록을 함께 표시

#### 결과

- 재생목록 레코드 생성: 소재 배치 순서(구좌) 포함
- 이 재생목록은 UC-22(편성표)에서 매체에 배정 가능

### UC-22: 편성표 생성

**누가**: 매체사 또는 운영대행사
**언제**: 재생목록을 특정 매체에 배정하고 재생 기간을 설정할 때
**전제조건**: UC-21로 재생목록 1개 이상 생성. 대상 매체가 UC-08로 디바이스 연동 완료 (연동됨(활성) 상태)

#### 시나리오

1. 매체사가 편성 관리 > 편성표 > 새 편성표 버튼을 클릭한다
2. 편성표명을 입력한다 (필수)
3. 재생목록을 선택한다 (필수)
4. 대상 매체 또는 매체 그룹을 선택한다 (복수 선택 가능, 최소 1개 필수)
5. 적용 기간을 설정한다 (시작일시·종료일시, 필수. 시·분까지 설정 가능. 기본값: 시작일 00:00, 종료일 23:59)
6. 캠페인을 연결한다 (선택, 드롭다운. 미연결 시 캠페인 취소 연쇄 종료 대상 아님)
7. 우선순위를 설정한다 (1~3등급, 기본값 3순위):
   - 3순위: 기본 편성 (디폴트)
   - 2순위: 특정 기간 덮어쓰기
   - 1순위: 촬영용/특수 목적 (소수 소재만 집중 송출)
   - 동순위 편성표 저장 불가. 반드시 우선순위를 다르게 설정하여 충돌 해소 (UC-24 연계)
   - 동일 매체에 4개 이상 편성표가 겹칠 때 3등급이 부족한 경우: 기간을 분할하여 겹침을 해소하거나, 기존 편성표를 종료한 후 새 편성표를 생성
8. 저장 버튼을 클릭한다
9. 시스템이 편성 충돌 여부를 검사한다 (UC-24 연계)
10. 충돌 없으면 편성표가 "예약됨" 상태로 생성된다
11. **"즉시 반영" 버튼** 클릭 시에만 편성 내용이 플레이어에 전달된다 (편성 수정 후 반드시 클릭 필요)

#### 이런 경우에는

- **비활성 매체 선택 시도**: "비활성 매체에는 편성표를 적용할 수 없습니다" 안내. 선택 불가
- **미연동 매체 선택 시도**: "디바이스 미연동 매체입니다" 안내. 선택 불가
- **시작일 > 종료일**: "시작일이 종료일보다 앞서야 합니다" 안내
- **동일 매체+동일 기간에 편성 충돌 (SCH-BR-03)**: UC-24로 이동하여 충돌 해소
- **운영대행사가 권한 범위 밖 매체 선택**: 해당 매체가 목록에 표시되지 않음
- **동시 편성표 수정 (낙관적 잠금)**: 편성표 조회 시 버전 번호를 함께 로드한다. 저장 시 로드 시점의 버전과 현재 DB 버전을 비교한다. 버전 불일치 시: "다른 사용자가 이 편성표를 수정했습니다. 최신 내용을 확인 후 다시 저장해주세요" 안내. 최신 편성표 내용을 새 창/패널에 표시하여 비교 가능. **재시도 3회 초과 시**: "편성표가 빈번하게 수정되고 있습니다. 잠시 후 다시 시도해주세요" 안내 후 저장 차단 (30초 쿨다운)

#### 결과

- 편성표 레코드 생성: 상태 "예약됨"
- 시작일 도래 시 자동 적용 (UC-25)

### UC-23: 네이버 Ad Server 연동 구좌 설정

**누가**: 어드민, 매체사 또는 운영대행사
**언제**: 재생목록 내 특정 구좌를 네이버 Ad Server와 연동하여 프로그래매틱 광고를 송출할 때
**전제조건**: UC-21로 재생목록 생성 완료. 해당 매체가 UC-09로 네이버 SSP 연동 완료

#### 시나리오

1. 어드민, 매체사 또는 운영대행사가 재생목록 상세로 진입한다
2. 특정 구좌를 선택한다
3. "네이버 Ad Server 연동" 토글을 활성화한다
4. 애드 슬롯 번호를 지정한다:
   - 100번대: 확장용
   - 200번대: 확장용
   - 300번대: 네이버 NAM 전용
5. 해당 구좌에 배치된 소재가 제거되고, "네이버 Ad Server" 라벨로 표시된다
6. 저장 버튼을 클릭한다
7. 이후 재생 시 해당 구좌는 네이버 Ad Server가 실시간으로 광고를 결정한다

> **데이터 흐름**: CMS → 플레이어 (편성정보 전달), NAM → 플레이어 (광고소재 직접 전달). CMS ↔ NAM 간 직접 소재 교환 없음

#### 이런 경우에는

- **매체에 네이버 SSP 연동이 안 된 상태**: "네이버 Ad Server 연동" 토글 비활성화. "먼저 매체의 SSP 연동을 설정해주세요" 툴팁
- **모든 구좌를 네이버 Ad Server 연동으로 설정**: 허용. 전체 프로그래매틱 운영 가능
- **네이버 Ad Server 연동 구좌를 다시 자체 소재로 전환**: 토글 해제 → 소재 선택 필요

#### 결과

- 해당 구좌가 네이버 Ad Server 연동 모드로 전환
- 나머지 구좌는 기존 순환 재생 방식 유지

### UC-24: 편성 충돌 감지 및 해소

**누가**: 어드민, 매체사 또는 운영대행사
**언제**: 편성표 생성/수정 시 동일 매체에 기간이 겹치는 편성이 감지될 때
**전제조건**: UC-22에서 편성표 저장 시 충돌 감지

#### 시나리오

1. 시스템이 편성표 저장 시 충돌을 감지한다:
   - 동일 매체 + 겹치는 기간 + 동일/낮은 우선순위
2. 충돌 상세를 표시한다:
   - 충돌 대상 편성표명, 기간, 우선순위
3. 해소 옵션을 제시한다:
   - **A)** 현재 편성표의 우선순위를 올려서 기존 편성표를 대체
   - **B)** 기존 편성표의 기간을 조정
   - **C)** 현재 편성표 저장을 취소
4. 사용자가 옵션을 선택하여 충돌을 해소한다

#### 이런 경우에는

- **3개 이상 편성표가 동시 충돌**: 모든 충돌 목록을 표시. 하나씩 순차 해소
- **우선순위가 동일한 편성표끼리 충돌**: 동순위 불허. 반드시 우선순위를 변경하여 해소해야 저장 가능

#### 결과

- 충돌 해소 후 편성표 저장 완료
- 편성 관리자에게 "편성 충돌 (SCH-BR-03)" 알림 발송 (UC-26 연계)

### UC-25: 편성표 적용/종료

**누가**: 시스템 (자동) 또는 매체사, 운영대행사 (수동 종료)
**언제**: 편성표 시작일이 도래하거나, 종료일이 경과하거나, 사용자가 수동 종료할 때
**전제조건**: UC-22로 편성표 생성 완료 ("예약됨" 상태)

#### 시나리오 A — 자동 적용

1. 시스템이 시작일이 도래한 편성표를 감지한다
2. 편성표 상태를 "예약됨" → "적용중"으로 변경한다
3. 해당 매체의 플레이어에 재생목록을 전송한다
4. 플레이어가 재생목록의 구좌를 순서대로 순환 재생한다
5. 편성 관리자에게 "편성 시작" 알림 발송

#### 시나리오 B — 자동 종료

1. 시스템이 종료일이 경과한 편성표를 감지한다
2. 편성표 상태를 "적용중" → "종료"로 변경한다
3. 편성 관리자에게 "편성 종료" 알림 발송

#### 시나리오 C — 수동 종료

1. 매체사 또는 운영대행사가 편성 관리 > 편성표 상세에서 "종료" 버튼을 클릭한다
2. 시스템이 확인 모달: "이 편성표를 즉시 종료하시겠습니까?"
3. 확인 시 편성표 상태 "적용중" → "종료"

#### 이런 경우에는

- **적용 시작 시 플레이어 오프라인**: 편성표는 "적용중"으로 전환되지만 실제 재생 불가. 매체 오프라인 알림 별도 발송 (UC-11 연계)
- **적용 시작일 전 대상 매체가 비활성화됨**: 편성표는 '예약됨' 상태를 유지한다. 편성 담당자에게 '대상 매체 비활성 — 시작일에 적용되지 않을 수 있습니다' 알림 발송. 시작일 도래 시 매체가 여전히 비활성이면 '예약됨' → '적용중' → '일시정지'로 전환한다
- **적용중 편성표의 재생목록을 수정**: 수정 후 저장만으로는 플레이어에 반영되지 않음. UC-22의 "즉시 반영" 버튼을 클릭해야 플레이어에 업데이트된 재생목록이 전송된다 (SCH-BR-10)
- **종료 후 재적용**: 불가. 새 편성표 생성 필요
- **편성표 vs 플레이어 상태 불일치**: CMS 편성표 상태는 "의도(intent)"이며, 플레이어 보고 상태는 "실제(actual)"이다. 플레이어는 현재 재생 중인 편성 버전을 주기적으로 CMS에 보고한다 (heartbeat). 의도와 실제가 불일치하면 편성표 상세에 "동기화 미완료" 배지를 표시한다. 불일치 지속 시 편성 담당자에게 알림 발송
- **동시 편성표 수정 (낙관적 잠금)**: UC-22와 동일하게 버전 기반 낙관적 잠금 적용. 버전 불일치 시 "다른 사용자가 이 편성표를 수정했습니다. 최신 내용을 확인 후 다시 저장해주세요" 안내
- **매체에 현재 적용 중인 편성표가 없음**: 플레이어는 "대기 화면"을 표시한다. 대기 화면은 매체사별로 설정 가능 (기본값: 시스템 기본 대기 이미지). CMS 매체 상세에서 "현재 편성 없음 — 대기 화면 송출 중" 상태를 표시한다
- **편성 관련 Slack 알림 발송 실패**: 발송 실패 시 해당 실패 사실을 인앱 알림으로 발송한다. 인앱 알림 내용: "Slack 알림 전송 실패 — 인앱에서 확인해주세요: [원본 알림 내용 요약]"
- **플레이어 전송 실패 재시도**: 플레이어가 온라인 상태인데 재생목록 전송이 실패하면, 지수 백오프로 최대 3회 재시도한다 (1분, 5분, 15분). 3회 모두 실패 시: 편성표 상세에 "동기화 미완료" 배지를 표시하고, 편성 담당자에게 알림 발송. 플레이어가 다음 heartbeat에서 버전 불일치를 보고하면 CMS가 재전송을 시도한다

#### 결과

- 적용: "예약됨" → "적용중". 매체에서 재생 시작
- 종료: "적용중" → "종료". 매체에서 재생 중단

### UC-32: 편성표 복제

**누가**: 매체사 또는 운영대행사
**언제**: 기존 편성표를 기반으로 날짜만 변경하여 빠르게 생성할 때
**전제조건**: 복제 원본 편성표가 존재

#### 시나리오

1. 매체사가 편성 관리 > 편성표 목록에서 기존 편성표를 선택한다
2. "복제" 버튼을 클릭한다
3. 새 적용 기간(시작일/종료일)을 지정한다
4. 시스템이 원본의 구좌 구조, 소재 배치, 우선순위를 모두 복제한다
5. 필요 시 소재만 부분 변경한다
6. 저장 버튼을 클릭한다

#### 이런 경우에는

- **원본 편성표의 소재가 삭제된 경우**: 복제 시 해당 구좌를 "삭제됨" 상태로 복제. "소재 삭제됨 — 교체 필요" 표시
- **원본 편성표의 소재가 만료된 경우**: 경고 표시. "만료 소재가 N건 포함되어 있습니다. 교체를 권장합니다"

#### 결과

- 새 편성표 레코드 생성: 원본의 구좌·소재·우선순위 복제, 기간만 변경
- 편성 충돌 검사 수행 (UC-24 연계)

#### 이런 경우에는

- **편성표에 "삭제됨" 구좌 포함**: 익스포트 파일에 "삭제됨" 상태 명시

### UC-34: 싱크 송출 (시간 동기화 강제 편성)

**누가**: 매체사 또는 운영대행사
**언제**: 매시 정각 등 지정 시각에 특정 광고를 강제 삽입할 때
**전제조건**: 대상 매체에 편성표가 적용중. "검수완료" 소재가 존재

#### 시나리오

1. 매체사가 편성 관리 > 싱크 송출 설정으로 진입한다
2. 대상 매체를 선택한다
3. 싱크 소재를 선택한다
4. 반복 시간을 설정한다 (시·분·초 단위)
5. 저장 버튼을 클릭한다
6. 해당 시각에 진행 중인 광고가 중단되고, 싱크 광고가 우선 송출된다
7. 싱크 광고 종료 후 다음 광고부터 재개된다

#### 이런 경우에는

- **중단된 광고의 처리**: 중단된 광고는 로그에서 미완료 처리. 기존 광고주의 계약 횟수(1일 100회 이상)는 별도 보장
- **복수 매체 동시 송출**: 허용. 사용 사례: 홍대 7개 매체 동시 싱크, 케이팝/아이돌 생일 이벤트, 영화 프로모션
- **싱크 송출과 일반 편성 충돌**: 싱크 송출이 일반 우선순위(1~3)보다 상위. 긴급 편성(UC-35)보다 하위

#### 결과

- 지정 시각마다 싱크 소재 강제 송출
- 기존 편성은 싱크 종료 후 재개
- 중단된 광고 로그 기록

### UC-35: 긴급 편성

**누가**: 어드민
**언제**: 국가 재난 방송, 소등 캠페인 등 모든 조건을 무시하고 즉시 송출할 때
**전제조건**: 어드민 권한. 송출 대상 소재 존재

#### 시나리오

1. 어드민이 편성 관리 > 긴급 편성을 클릭한다
2. 대상 매체를 선택한다 (전체 또는 특정 매체)
3. 긴급 소재를 선택한다
4. "긴급 송출 시작" 버튼을 클릭한다
5. 시스템이 확인 모달: "모든 기존 편성/싱크를 무시하고 긴급 소재만 송출합니다. 계속하시겠습니까?"
6. 확인 시 지정 소재만 즉시 송출 시작 (모든 기존 편성/싱크 무시)
7. "긴급 송출 종료" 버튼으로 해제

#### 이런 경우에는

- **긴급 편성 중 싱크 송출 시각 도래**: 싱크 송출 무시. 긴급 편성이 최고 우선순위
- **긴급 편성 해제 후**: 기존 편성/싱크가 자동 재개

#### 결과

- 최고 우선순위로 지정 소재만 송출
- 모든 기존 편성/싱크 일시 무시
- 해제 시 기존 편성 자동 복귀

## 5. 도메인 규칙

### 5.1 편성표 상태 FSM

```
[예약됨] ──(시작일 도래)──→ [적용중] ──(종료일 경과 또는 수동 종료)──→ [종료]
                               │        ↑
                          (매체 비활성화) (매체 활성화 + 종료일 미경과)
                               ↓        │
                           [일시정지] ───┘
                               │
                          (종료일 경과)──→ [종료]
```

**상태**:
| 상태 | 설명 | 허용 동작 |
|------|------|----------|
| 예약됨 | 편성표가 생성되어 저장된 상태. 적용 시작일 도래 전 | 수정, 삭제 |
| 적용중 | 매체에서 재생 중 | 재생목록 수정, 수동 종료 |
| 일시정지 | 매체 비활성화로 송출 중단, 매체 활성화 시 자동 재개 가능 | 조회, 매체 활성화 대기 |
| 종료 | 재생 완료/중단 | 조회만 |

**전이**:
| From → To | 트리거 | 가드 | 효과 |
|-----------|--------|------|------|
| 예약됨 → 적용중 | 시작일 도래 | 대상 매체 활성 상태 | 플레이어에 재생목록 전송, 알림 |
| 적용중 → 종료 | 종료일 경과 (자동) | — | 재생 중단, 알림 |
| 적용중 → 종료 | 사용자가 수동 종료 | 권한 보유 | 즉시 재생 중단, 알림 |
| 적용중 → 종료 | 연결 캠페인 취소 (UC-19) | 캠페인 FK 연결됨 | 연쇄 종료, 편성 담당자 알림 |
| 예약됨 → 종료 | 연결 캠페인 취소 (UC-19) | 캠페인 FK 연결됨 | 종료 처리, 편성 담당자 알림 |
| 적용중 → 일시정지 | 대상 매체 비활성화 | — | 송출 중지. 비활성화 주체 ≠ 편성 생성자이면 생성자에게 알림 (UC-12 참조) |
| 일시정지 → 적용중 | 대상 매체 활성화 | 종료일 미경과 | 플레이어에 재생목록 재전송, 송출 자동 재개 |
| 일시정지 → 종료 | 종료일 경과 (자동) | — | 종료 처리. 매체 활성화 시에도 복구 불가 |

> 매체 상태(활성/비활성)와 편성표 상태의 연계는 매체 관리 UC-12를 참조한다.
> 매체 비활성화 시 "적용중" → "일시정지", 매체 활성화 시 종료일 재검사 후 복귀.

**불변식**:
- INV-18: 편성표는 최소 1개의 재생목록을 가져야 함
- INV-19: 편성표는 최소 1개의 대상 매체를 가져야 함
- INV-20: 비활성 매체에 편성표 적용 불가
- INV-21: 재생목록의 구좌는 1개 이상이어야 함 (빈 재생목록 불가)

### 5.2 구좌 상태

| 상태 | 설명 | 재생 시 동작 | UI 표시 |
|------|------|------------|---------|
| 정상 | 소재가 배정된 구좌 | 소재 재생 | 기본 표시 |
| 삭제됨 | 배정 소재가 삭제된 구좌 | 스킵 (구좌 구조 보존) | 빨간 점선 + "소재 삭제됨 — 교체 필요" |
| 교체중 | 배정 소재의 파일 교체 후 재검수 진행 중 | 기존 파일로 재생 유지 | "교체중" 라벨 |
| 더미(스킵) | 분할 구좌용 더미 구좌 | 스킵 (구좌 구조 보존) | "더미" 라벨 (회색 배경) |
| 네이버 Ad Server 연동 | 네이버 Ad Server가 소재를 결정하는 구좌 | Ad Server 실시간 결정 | "네이버 Ad Server" 라벨 |

> `삭제됨`, `더미(스킵)`, `네이버 Ad Server 연동`은 모두 자체 소재가 없는 구좌이나, 동작과 표시가 다르므로 명확히 구분한다.

### 5.3 재생 모델

- **기본**: 순환 재생 — 재생목록의 구좌를 순서대로 무한 반복
- **확장**: 네이버 Ad Server 연동 구좌는 Ad Server가 실시간 결정
- 나머지 구좌는 기존 순환 재생 유지
- 별도 LoopPolicy 엔티티 없음

### 5.4 편성 우선순위 체계

```
긴급 편성 > 싱크 송출 > 1순위 > 2순위 > 3순위
```

- 긴급 편성(UC-35): 최고 우선순위. 모든 편성/싱크 무시. 어드민 전용
- 싱크 송출(UC-34): 일반 우선순위(1~3)보다 상위, 긴급보다 하위
- 1순위: 촬영용/특수 목적 (소수 소재 집중 송출)
- 2순위: 특정 기간 덮어쓰기
- 3순위: 기본 편성 (디폴트)

### 5.5 비즈니스 규칙

- SCH-BR-01: 구좌의 재생 시간 = 배정된 소재의 재생 시간 (구좌 자체 시간 설정 없음)
- SCH-BR-02: 동일 매체에 복수 편성표 적용 가능. 우선순위로 결정
- SCH-BR-03: 편성 충돌 = 동일 매체 + 겹치는 기간 + 우선순위 미결정
- SCH-BR-04: 네이버 Ad Server 연동 구좌는 SSP 연동 완료 매체에만 설정 가능
- SCH-BR-05: 편성표 상태 자동 전환은 시작일시·종료일시를 분 단위로 지정하며, 해당 시각에 즉시 전환한다
- SCH-BR-06: 우선순위는 1~3 등급. 기본값 3
- SCH-BR-07: 긴급 편성은 모든 우선순위를 무시하는 최상위 등급 (어드민 전용)
- SCH-BR-08: 싱크 송출은 일반 우선순위(1~3)보다 상위, 긴급보다 하위
- SCH-BR-09: 더미 구좌는 재생 시 스킵. 분할 구좌 구현에 사용
- SCH-BR-10: 편성 수정 후 "즉시 반영" 클릭 시에만 플레이어에 전달
- SCH-BR-11: 동일 매체 + 겹치는 기간에 동순위 편성표 저장 불가. 우선순위 1~3등급이므로 **동일 매체 × 동일 시간 구간에 최대 3개 편성표**만 공존 가능. 충돌 판정은 1분이라도 겹치면 적용. 4개 이상 필요 시: (a) 기간을 분할하여 겹치는 구간을 줄이거나, (b) 기존 편성표를 종료한 후 새 편성표를 생성

## 6. 미결 사항

- ~~편성표 우선순위의 상세 동작 미정의~~ → 1~3등급 + 싱크 + 긴급 체계로 확정 (5.4 편성 우선순위 체계 참조)
- 재생목록 수정 시 적용중 편성표에 즉시 반영되는 메커니즘 상세 미정의
- 종료된 편성표 이후 매체에 표시할 기본 콘텐츠(대기 화면) 미정의
- 편성표 없는 매체의 기본 동작 미정의 (블랙스크린? 기본 이미지?)
- 재생목록 삭제 시 해당 재생목록을 참조하는 편성표 처리 미정의
- 네이버 Ad Server 응답 지연/실패 시 해당 구좌의 fallback 동작 미정의
- 플레이어에 재생목록 전송 실패 시 재시도 정책 미정의
- 네이버 Ad Server 연동 구좌와 자체 구좌 혼합 시 전체 순환 재생 시간 계산 방식 미정의 (네이버 구좌 시간은 가변)
- **[인터뷰 도출]** UC-24 편성 충돌 감지 시점 — 현재 저장 후 감지이나, 인터뷰에서 매체사 팀장이 "저장하고 나서야 충돌을 알게 된다"는 패턴 확인됨. 저장 전 실시간 충돌 경고(입력 중 감지) 방식의 구현 타당성 검토 필요
- **[인터뷰 도출]** 편성 타임라인 시각화 미정의 — 매체사 팀장이 CMS 편성 화면만으로 전체 일정 파악이 안 되어 엑셀 병행 관리 중. 매체별 편성표를 간트 차트/타임라인 뷰로 표시하는 기능 요구 확인됨
- **[인터뷰 도출]** 재생목록-편성 2단계 구조의 온보딩 가이드 미정의 — 운영대행사 현장 매니저가 재생목록을 먼저 만들어야 편성이 가능하다는 구조를 파악하지 못해 편성 누락 발생. 첫 편성 생성 시 단계 안내 또는 인컨텍스트 도움말 필요
- "삭제됨" 구좌 스킵 시 플레이어 에러 핸들링 상세 — 플레이어 설계 단계에서 정의
- 매체 비활성화 신호 전파 중 일시적 상태 불일치 허용 범위 — 구현 단계에서 정의
- 싱크 송출 초(second) 단위 설정의 정밀도 한계 미정의
- 긴급 편성 발동/해제 프로토콜 상세 미정의
- 편성표 익스포트 파일 형식 미정의 (Excel? PDF?)
