# 리서치 종합 인사이트

생성일: 2026-04-05
대상 페르소나: 어드민(박진수), 매체사(김서영), 운영대행사(이준혁), 영업대행사(최유진)

---

## 1. 핵심 발견 (Cross-Persona Themes)

### 테마 1: CMS 상태와 현장 실제의 불일치
- **무엇**: CMS가 "적용중"이어도 현장 플레이어는 구버전을 재생하고 있는 경우가 일상적으로 발생
- **누가 겪는가**: 매체사, 운영대행사, 어드민 (영업대행사는 2차 피해)
- **왜 중요한가**: 광고주 분쟁, 무상 집행 보상, 플랫폼 신뢰 훼손으로 직결
- **설계 함의**: 매체별 "의도한 편성 vs 현재 재생 중인 편성" 이원 표시 필수. 동기화 미완료 건수를 대시보드 최상위에 노출

### 테마 2: 검수 블랙박스가 모든 역할의 병목
- **무엇**: "검수중" 이후 언제 끝나는지, 왜 실패했는지, 기준이 뭔지를 아무도 모름
- **누가 겪는가**: 영업대행사(광고주 약속 불이행), 운영대행사(급한 편성 지연), 매체사(검수 대기 압박)
- **왜 중요한가**: 온에어 일정 실패 → 광고주 클레임 → 계약 리스크
- **설계 함의**: 검수 예상 소요 시간 + 단계별 진행률 표시. 실패 사유는 비기술적 언어로 구체적 기재("해상도 불일치: 1920×1080 필요, 1280×720 업로드됨"). 업로드 전 매체별 규격 사전 안내

### 테마 3: 동시 편집으로 인한 작업 소실
- **무엇**: 여러 사용자가 같은 편성표를 동시에 수정할 때 나중에 저장한 쪽이 앞의 작업을 덮어씀
- **누가 겪는가**: 운영대행사(30분 작업 소실 경험), 매체사(대행사와 동시 작업)
- **왜 중요한가**: 월 1~2회 발생. 광고주 미방영 사고로 이어짐
- **설계 함의**: 저장 전 낙관적 잠금 충돌 경고보다 **사전 경고**("지금 김 대리님이 이 편성표를 열람 중") 우선. 충돌 발생 시 diff 비교 패널 제공

### 테마 4: 알림이 도달하지 않거나 노이즈에 매몰
- **무엇**: 긴급 알림이 인앱·브라우저 푸시·Slack 채널 장애로 도달 실패하거나, 새벽 업데이트 알림에 묻혀 무시됨
- **누가 겪는가**: 어드민(Slack 조용한 실패), 운영대행사(알림 과부하), 영업대행사(외근 중 인앱 미수신)
- **왜 중요한가**: 장애 감지 지연 → 매체사 문의 후에야 인지 → 대응 골든타임 소실
- **설계 함의**: 알림 채널 헬스체크(30분 주기 Slack 핑). 우선순위별 채널 분리(긴급 → 모바일 SMS/카카오, 일반 → 인앱). 어드민 긴급 알림에 이메일/SMS fallback 필수

### 테마 5: "검수완료 = 송출중"이라는 오해
- **무엇**: 검수완료 시점에 영업대행사가 광고주에게 "반영됐습니다" 보고하지만, 편성 연결 전이면 실제 미송출
- **누가 겪는가**: 영업대행사(광고주 신뢰 파탄), 운영대행사(교체 즉시 반영 오인)
- **왜 중요한가**: 광고주가 현장에서 미송출 확인 시 대행사 신뢰 붕괴
- **설계 함의**: 소재 상세에 편성 연결 상태 읽기 전용 표시 — "미편성 / 편성대기 / 송출중 / 송출종료" 4단계. 영업대행사 접근 범위 내에서 소재→편성 상태 연계 조회 허용

### 테마 6: 운영 컨텍스트 관리 부재 (멀티-엔티티 피로)
- **무엇**: 운영대행사는 3개 매체사 계정을 브라우저 탭 분리로 관리. 컨텍스트 전환 시 실수 발생
- **누가 겪는가**: 운영대행사(탭 전환 혼동), 영업대행사(40~50개 소재 개별 클릭 확인)
- **왜 중요한가**: 멀티 엔티티 관리 특성이 UI에 반영되지 않아 인간 에러 유발
- **설계 함의**: 역할별 맞춤 대시보드 필수. 운영대행사: 매체사별 편성 현황 요약. 영업대행사: 소재 상태 요약 위젯(검수중/완료/실패/만료임박)

### 테마 7: 정산·이력 추적을 위한 감사 로그 부재
- **무엇**: "누가 언제 편성을 바꿨는지", "어떤 소재가 언제까지 나갔는지" 추적 수단 없음
- **누가 겪는가**: 매체사(정산 분쟁 대응 불가), 어드민(장애 원인 추적 불가)
- **왜 중요한가**: 분쟁 발생 시 증거 부재 → 무상 보상으로 귀결
- **설계 함의**: 편성 수정 이력(누가/언제/어떤 변경)과 플레이어 재생 이력 로그를 정산 리포트와 연계

## 4. 위험 요소 (Design Risks)

1. **동기화 상태를 이진(온라인/오프라인)으로만 표시하면**: "온라인이면 정상 재생 중"이라는 오해가 고착화. 5단계 구분(온라인-정상 / 온라인-동기화 미완료 / 오프라인 / 이상 감지 / 비활성) 필요
2. **검수 실패 사유를 기술 용어로 표시하면**: 운영대행사·영업대행사가 이해하지 못해 제작사 전달 과정에서 정보 왜곡. 비기술적 언어 + 공유용 텍스트 복사 기능 없으면 캡쳐→카톡 우회 시 타 광고주 정보 노출 보안 사고 가능
3. **소재 상태를 검수완료에서 끝내면**: 영업대행사가 광고주에게 오보고. "검수완료 ≠ 송출중" 갭이 신뢰 이탈로 직결
4. **알림을 단일 채널(Slack/인앱)에만 의존하면**: 채널 장애 시 모든 긴급 알림이 소실. 특히 Slack Webhook 만료는 조용히 실패하여 감지 불가 상태로 장기 방치
5. **편성 충돌 해소 UI가 기술적 우선순위 용어를 그대로 노출하면**: 기술 이해도 낮은 운영대행사가 판단을 포기하고 매체사에 전화 위임 → CMS 자체 해소율 하락

## 5. 미해결 질문

- **원격 스크린샷 기술 실현 가능성**: 플레이어에서 주기적 스크린샷 CMS 전송 가능한지 플레이어 팀 확인 필요 (매체사·운영대행사 모두 강하게 요구)
- **SMS/카카오 알림톡 채널 추가 비용·기술 범위**: 영업대행사 외근 패턴 대응에 필수이나 06 알림 시나리오 전반 재설계 필요. 건당 과금·발송 대상 전화번호 수집 정책 포함
- **소재 버전 관리 스토리지 정책**: 영상 파일 특성상 버전당 수백 MB. 보존 버전 수·기간 제한 정책과 비용 검토 필요
- **소재→편성 연결 상태 노출 수준**: 영업대행사에게 매체명만 보여줄지, 편성 기간/시간대까지 보여줄지. 05 편성 시나리오 담당자와 접근 범위 경계 합의 필요
- **재생목록-편성표 2단계 구조 단순화 가능성**: 운영대행사 다수가 "편성표에서 바로 소재 넣으면 안 되나?"로 혼란. 2단계 구조 유지 근거를 설계에서 시각적으로 납득시키는 방법 탐색 필요
- **긴급 편성 복귀 시 원래 편성표 종료일 경과 처리**: UC-35 미정의. 긴급 발동 중 원본 편성 종료일 도래 시 복귀 후 재생 상태 정의 필요
