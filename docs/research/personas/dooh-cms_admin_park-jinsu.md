# 박진수 (어드민)

## 기본 정보
- **이름**: 박진수
- **나이**: 38세
- **직책**: CMS 플랫폼 관리자 (IT 운영 10년차)
- **역할**: 어드민
- **한 줄 요약**: "이거 운영 중에 터지면 어떡해요?"

## 인구통계
- 기술적 이해도: **높음** (API, DB, 서버 로그 직접 확인 가능)
- IT 시스템 운영 경력 10년, 옥외광고 도메인 3년차

## 목표
- **기능적**: 전체 시스템 안정 운영, 장애 빠른 감지·대응
- **감정적**: 장애가 터져도 "원인 파악 → 대응 → 보고"가 30분 내 끝나는 안심감
- **사회적**: 매체사·대행사에게 "이 플랫폼 믿고 쓸 수 있다" 신뢰 제공

## 고충 (Frustrations)
- 장애 발생 시 원인 파악이 느림 (CMS 상태와 현장 실제가 다를 때 특히)
- 매체사/대행사에게 상황 전달이 지연됨
- Slack 알림이 조용히 실패하면 장애 자체를 모름 ("조용한 장애")
- 매체 삭제 기능 부재로 고아 데이터 누적

## 행동 패턴
- 매일 수십 개 매체사의 매체 상태 모니터링
- 장애 대응 1차 담당자로 Slack + 인앱 알림 상시 확인
- API, DB 직접 조회로 상태 불일치 추적
- 주기적으로 시스템 헬스체크 결과 리뷰

## 시나리오 (Day-in-the-Life)
오전 9시 출근 후 Slack 장애 채널 확인. 매체 상태 대시보드에서 오프라인 매체 없는지 훑어봄. 오전 중 매체사 온보딩 1건 처리 (UC-06). 오후에 매체사로부터 "편성 반영 안 되는 것 같다" 문의 — CMS에서는 "적용중"인데 현장 확인 요청. 플레이어 로그 추적해서 재동기화 문제 확인. 퇴근 전 알림 실패 로그 점검.

## 디자인 시사점
- 알림 채널 health check 기능 필수 (Slack 실패 감지)
- CMS 상태와 플레이어 실제 상태 불일치 시각화 (배지/경고)
- 장애 원인 추적을 위한 감사 로그·이력 쉽게 접근 가능해야 함
- 시스템 전체 상태를 한눈에 볼 수 있는 운영 대시보드
