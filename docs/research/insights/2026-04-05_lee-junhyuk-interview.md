## 페르소나 정의와 다른 응답

- **"즉시 반영" 인지 부재**: 페르소나 정의에서는 "UI만 사용"이라고 했으나, 실제로는 "즉시 반영" 버튼의 존재 자체를 인지하지 못하는 수준. 편성표 저장 = 현장 반영이라고 생각하고 있었음. "버튼이 더 있었어요?"라는 반응. 기술적 이해도가 "낮음"보다 더 구체적으로 — **CMS의 2단계 적용 모델(저장 vs 반영)을 이해하지 못하는 수준**으로 재정의 필요
- **실수 빈도**: "실수하기 쉬움"이라는 정의보다 심각. 월 2~3회 편성 사고 발생 중. 사고 후 매체사에서 직접 전화가 오는 패턴이 루틴화됨
- **자체 판단 회피 정도**: 정의에서 "편성 충돌 시 매체사에 확인 요청"이라고 했으나, 실제로는 충돌뿐 아니라 **우선순위 변경, 기간 조정 등 모든 의사결정을 매체사에 위임**. 본인 권한으로 해소 가능한 건도 매체사 확인 후 진행

## 신규 발견

### I-10. "즉시 반영" 2단계 모델의 인지 실패 [상]
- **현상**: 이준혁은 편성표 저장 후 "즉시 반영" 버튼을 누르지 않아 편성이 현장에 적용되지 않는 사고를 반복. "저장했는데 왜 안 나가요?"라고 매체사에 문의 → 매체사가 "즉시 반영 눌렀어?"라고 역질문. 시작일 자동 적용(UC-25)과 "즉시 반영"(SCH-BR-10)의 차이를 전혀 이해하지 못함
- **관련 도메인**: 05-편성관리
- **심각도**: 상
- **시나리오 미반영**: UC-22 시나리오에 "즉시 반영을 누르지 않았을 때 어떻게 되는가"에 대한 이런 경우에는(예외 흐름)가 없음. 저장 후 즉시 반영을 누르지 않으면 편성표 상태는 "예약됨"으로 유지되나, 사용자는 이것을 "적용 완료"로 오해. **저장 후 즉시 반영 미클릭 시 안내 메시지 또는 배지("미반영") 표시 필요**
- **제안**: (1) 저장 후 "편성 내용이 저장되었습니다. 플레이어에 반영하려면 '즉시 반영'을 클릭하세요" 토스트 안내, (2) 편성표 목록에 "미반영" 배지 추가, (3) 첫 사용자 온보딩에서 2단계 모델 설명

### I-11. 디바이스 교체 실패 시 복구 경로 불명확 [상]
- **현상**: 장비 고장 → 현장 기사가 새 플레이어 설치 → CMS에서 "디바이스 교체" 클릭 → 새 디바이스 ID 입력 → 연동 테스트 실패(네트워크 문제) → 매체 상태 "미연동"으로 전이 → **기존 편성표가 "일시정지"** → 블랙스크린 → 광고주 클레임. 이 상황에서 이준혁은 "미연동 상태에서 다시 연동 시도" 버튼을 찾지 못하고, 매체사에 전화해서 "어떻게 해요?"라고 물어봄
- **관련 도메인**: 02-매체관리
- **심각도**: 상
- **시나리오 미반영**: UC-08에 교체 실패 시나리오는 있으나, **실패 후 복구 UI 경로**가 불명확. "재시도하거나 다른 디바이스 ID를 입력해주세요"라는 안내만 있고, 실제 화면에서 어떤 상태로 보이는지, 편성표가 일시정지된 것을 어떻게 인지하는지가 없음
- **제안**: (1) 교체 실패 시 매체 상세에 "디바이스 미연동 — 재연동 필요" 배너 + 재연동 버튼 상시 표시, (2) 편성표 목록에서 해당 매체의 편성이 "일시정지(디바이스 미연동)" 사유와 함께 표시, (3) 교체 실패 시 자동 알림 발송(편성 담당자 + 매체사)

### I-12. 4개 이상 편성 겹침 시 "기간 분할" 가이드 부재 [중]
- **현상**: 3개 매체사 위탁 관리 중 하나의 매체에 4~5개 편성표가 겹치는 상황 빈번. 우선순위 3등급이 부족하여 "기간을 분할하여 겹침을 해소"해야 하는데, 구체적으로 어떻게 분할하는지 모름. 결국 기존 편성표를 종료하고 새로 만들면서 기간 앞뒤를 수동으로 쪼갬 → 편성표 수가 2배로 증가 → 관리 복잡도 폭증
- **관련 도메인**: 05-편성관리
- **심각도**: 중
- **시나리오 미반영**: UC-22 step 7에 "기간을 분할하여 겹침을 해소"라고만 기술. **분할 도구나 가이드가 시나리오에 없음**
- **제안**: (1) 충돌 해소 UI(UC-24)에서 "기간 분할" 옵션 선택 시 시스템이 자동으로 비겹침 구간을 계산하여 제안, (2) 분할 후 편성표가 2개로 나뉘는 것을 시각적으로 미리보기

### I-13. 낙관적 잠금 충돌 시 작업 내용 유실 [상]
- **현상**: 매체사 A 담당자와 동시에 같은 편성표를 30분간 수정. 저장 시 "다른 사용자가 수정했습니다" 안내 → 본인 작업 전체 유실. "비교 가능"이라고 했지만 실제로는 30분간의 구좌 재배치를 다시 해야 함. "비교해봤자 다시 해야 하는 건 똑같아요"
- **관련 도메인**: 05-편성관리
- **심각도**: 상
- **시나리오 미반영**: UC-22 낙관적 잠금 시나리오에 "최신 내용을 새 창/패널에 표시하여 비교 가능"까지만 기술. **충돌 시 본인 변경사항을 임시 저장하거나 머지하는 방법이 없음**
- **제안**: (1) 충돌 감지 시 본인 변경사항을 "임시 저장본"으로 자동 보존, (2) 최신 버전과 본인 버전을 나란히 비교 + 본인 변경사항을 선택적으로 적용(머지) 가능, (3) 편성표 수정 진입 시 "현재 다른 사용자가 수정 중입니다" 실시간 표시(편집 진입 경고)

### I-14. 네이버 Ad Server 구좌 fallback 동작 미인지 [중]
- **현상**: 네이버 연동 구좌에서 응답이 없으면 "까만 화면이 나간다". 이준혁은 이것이 CMS 문제인지 네이버 문제인지 구분할 수 없음. 현장에서 할 수 있는 일이 없어서 매체사에 전화 → 매체사도 모름 → 어드민에 에스컬레이션. 30분~1시간 블랙스크린 지속
- **관련 도메인**: 05-편성관리, 02-매체관리(SSP 연동)
- **심각도**: 중
- **시나리오 미반영**: 05 미결사항에 "네이버 Ad Server 응답 지연/실패 시 fallback 동작 미정의"로만 기재. 실무에서는 이미 발생 중인 문제
- **제안**: (1) 네이버 구좌 응답 실패 시 fallback 소재(기본 이미지 또는 다음 자체 구좌)로 자동 전환, (2) 응답 실패 이벤트를 CMS 대시보드에 실시간 표시 + 알림 발송, (3) 플레이어 단에서 타임아웃(예: 5초) 후 자동 fallback

### I-15. 매체사 전환 시 컨텍스트 혼동 [중]
- **현상**: 브라우저 탭 3개로 3개 매체사를 동시 작업. 탭 제목이 모두 "CMS - 편성관리"로 동일 → 매체사 B의 편성을 매체사 A 탭에서 작업하는 실수. 매체사 구분 없이 편성표 이름만 보고 작업하다가 잘못된 매체사에 저장. 특히 매체 이름이 비슷한 경우(예: "강남역 1번 출구", "강남역 2번 출구"가 서로 다른 매체사 소속) 혼동
- **관련 도메인**: 공통 UX
- **심각도**: 중
- **시나리오 미반영**: 시나리오 전반에 매체사 간 전환 UX가 정의되지 않음
- **제안**: (1) 브라우저 탭 제목에 매체사명 포함("CMS - [시티미디어] 편성관리"), (2) 매체사별 컬러 코드(헤더 색상 등)로 시각적 구분, (3) 편성표 저장 시 "이 편성은 [매체사 A]에 저장됩니다" 확인 표시

### I-16. 편성표 수정 이력 부재 [하]
- **현상**: 덮어쓰기 사고나 편성 오류 후 "누가 언제 뭘 바꿨는지" 추적 불가. 매체사에서 "왜 편성이 바뀌었냐"고 물으면 답할 수 없음
- **관련 도메인**: 05-편성관리
- **심각도**: 하
- **시나리오 미반영**: 편성표 수정 이력(audit log) 시나리오가 없음
- **제안**: 편성표 상세에 수정 이력 탭 추가 (시각, 수정자, 변경 내용 diff). 1차는 조회 전용으로 충분

## 추가 리서치 필요 영역

- **"즉시 반영" 워크플로우 재설계 검토**: 2단계 모델(저장 → 반영)이 저기술 사용자에게 근본적으로 맞지 않을 가능성. 저장 시 자동 반영(옵트아웃 방식)으로 전환할지, 현재 모델을 유지하되 UX 보강할지 → 매체사·어드민 의견 추가 수집 필요
- **편성 우선순위 등급 확장**: 3등급으로 부족한 상황이 운영대행사에서 빈번. 5등급 확장 또는 소수점 우선순위(예: 2.5순위) 도입 검토 → 전체 이해관계자 영향도 분석 필요
- **네이버 Ad Server fallback 정책**: 플레이어 팀과 네이버 측 협의 필요. 타임아웃 기준, fallback 소재 선정 로직, 장애 복구 후 자동 전환 여부
- **편성표 동시 편집 UX**: 낙관적 잠금 + 비교 뷰를 넘어, 실시간 공동 편집(Google Docs 방식)까지 검토할지, 편집 잠금(비관적 잠금)으로 전환할지 → 동시 편집 빈도 데이터 필요
- **운영대행사 전용 대시보드**: 3개 매체사를 한 화면에서 관리하는 "크로스 매체사 뷰" 필요성 → 권한 모델과의 충돌 여부 검토 필요
