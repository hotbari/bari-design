# 인터뷰 인사이트 — 최유진 (영업대행사 AE)

**일시**: 2026-04-05
**페르소나**: 최유진 (31세, 브라이트미디어 영업팀, 영업대행사)
**방식**: 페르소나 시뮬레이션 인터뷰
**목적**: 소재 관리 및 광고주 커뮤니케이션 관점에서 시나리오 미결 사항 검증

## 페르소나 정의와 다른 응답

- **인앱 알림 비활용**: 페르소나의 행동 패턴 "검수 상태 수시 확인(새로고침 반복)"은 맞지만, 인앱 알림 자체는 거의 확인하지 않는다고 응답. CMS에 상주하지 않는 AE 업무 특성 반영 필요 — 알림 채널이 인앱 중심이면 영업대행사에게 도달하지 못함
- **보고 타이밍 오해**: 페르소나에 "광고주에게 '정상 반영됐습니다' 보고 후 실제 송출은 확인 불가"로 기재했으나, 실제로는 "검수완료 = 송출중"으로 광고주가 오해하는 문제가 더 크다. 검수완료와 편성/송출 사이의 상태 갭이 페르소나 정의보다 심각
- **만료 알림 리드타임**: 페르소나 보정에서 "7일 전"으로 수정했으나, 인터뷰에서는 "10일 전도 괜찮다"고 응답 — 실무 리드타임이 예상보다 김 (광고주 의사결정 + 제작 + 검수)

## 신규 발견

### I-1. 검수완료↔송출중 상태 갭에 의한 광고주 오보고 [상]
- **현상**: AE가 "검수완료" 시점에 광고주에게 "반영됐습니다" 보고. 그러나 편성이 아직 안 된 상태이므로 실제 송출 안 됨. 광고주가 현장에서 미송출 확인 시 신뢰 깨짐
- **문제**: 소재 도메인에서 검수완료 이후 상태(편성됨, 송출중)가 보이지 않아 AE가 정확한 보고 불가. 현재 시나리오에 "소재→편성 연결 상태 표시"가 없음
- **관련**: 03 소재 관리 ↔ 05 편성 관리 교차, INV-13(영업대행사 소재 도메인 제한)
- **제안**: 소재 상세 화면에 편성 연결 상태를 읽기 전용으로 표시 — "미편성 / 편성대기 / 송출중 / 송출종료" 단계. 기존 발견 #2(송출 매체 목록 뷰)와 통합 설계 검토

### I-2. 수동검수대기 전환 인지 실패 — 알림 채널 부적합 [상]
- **현상**: 자동검수→수동검수대기 전환 시 인앱 알림만 발송(UC-14). AE는 CMS에 상주하지 않아 인앱 알림 미확인. "검수중"과 "수동검수대기" 차이를 인지 못한 채 금요일→월요일까지 방치
- **문제**: 수동검수대기가 금요일 전환 시 주말 동안 14일 타임아웃까지 허비. 광고주 온에어 일정 차질
- **관련**: 03 UC-14 이벤트 카탈로그, 06 알림 채널
- **제안**: 수동검수대기 전환 알림의 우선순위를 "일반→긴급"으로 상향하여 브라우저 푸시 추가. 장기적으로 모바일(SMS/카카오 알림톡) 채널 검토

### I-3. 매체 선택 오류 미감지 — 편성 시점까지 발견 지연 [중]
- **현상**: 일괄 업로드 시 매체 선택을 잘못하면 검수는 통과하지만(해상도 자동 검수가 "대상 해상도" 메타데이터 기준이므로 매체별 실제 해상도와 불일치 가능), 편성 시점에 가서야 문제 발견. 영업대행사는 편성 화면 접근 불가로 매체사 연락 전까지 인지 못함
- **문제**: UC-13의 "대상 매체 선택(필수)"과 UC-14의 해상도 검증 간 연결이 약함. 매체가 요구하는 실제 해상도와 업로드된 파일 해상도를 교차 검증하지 않는 상태
- **관련**: 03 UC-13 메타데이터 입력, UC-14 해상도 검증
- **제안**: UC-14 자동 검수에 "선택된 대상 매체의 실제 해상도와 파일 해상도 일치 여부" 검증 항목 추가. 매체별 해상도 마스터 데이터 필요

### I-4. 소재 교체 이력 미보존 — 이전 버전 롤백 불가 [중]
- **현상**: 광고주가 신규 소재 반응 부진 시 이전 버전 복원 요청. 현재 교체 시 이전 파일 소실(UC-16 시나리오 A: "파일만 교체"). AE가 로컬 PC에 이전 버전 수동 보관하는 우회 패턴
- **문제**: 03 미결 사항에 "소재 버전 관리 필요 여부 미정의"로 열려 있으나, 실무에서 롤백 수요 확인됨. 출장/재택 시 로컬 파일 접근 불가로 업무 중단
- **관련**: 03 UC-16 미결 "소재 버전 관리(교체 이력 보존) 필요 여부"
- **제안**: 소재 교체 시 이전 버전 자동 보존(최대 3~5버전). 소재 상세에서 버전 이력 조회 + "이 버전으로 되돌리기" 기능. 스토리지 보존 기간은 기존 soft delete 정책(90일)과 동일 적용

### I-5. 영업대행사용 소재 현황 대시보드 부재 [중]
- **현상**: AE가 40~50개 소재를 담당하나, 한눈에 상태를 파악할 수 없음. 소재 목록에서 하나하나 클릭해야 함. 아침 출근 시 "오늘 뭘 챙겨야 하는지" 파악에 시간 소요
- **문제**: 현재 07 대시보드 시나리오는 매체사/운영대행사 관점(매체 상태, 편성 현황 위주). 영업대행사가 소재 도메인만 접근하는 상황에서 소재 중심 대시보드 미설계
- **관련**: 07 대시보드, INV-13 접근 범위 제한
- **제안**: 영업대행사 로그인 시 소재 현황 요약 위젯 — 상태별 건수(검수중/완료/실패), 만료 임박 소재, 검수실패 소재 하이라이트. 기존 대시보드의 역할별 위젯 구성에 포함

### I-6. 검수 실패 캡쳐 시 타 소재 정보 노출 위험 [중]
- **현상**: 검수 실패 사유를 화면 캡쳐→카톡 전달 시, 캡쳐 범위에 다른 광고주 소재 정보(소재명, 광고주명)가 함께 노출될 수 있음. 경쟁 광고주 정보 유출 위험
- **문제**: 기존 발견 #1(복사 버튼)은 편의성 관점이었으나, 캡쳐 방식에는 정보 보안 위험도 내재. 특히 AE가 복수 광고주를 담당하는 구조에서 심각
- **관련**: 03 UC-14 검수 실패 사유 표시 UI
- **제안**: 검수 실패 사유 + 수정 가이드의 "공유용 텍스트 복사" 버튼 구현 시, 해당 소재 정보만 포함된 클린 텍스트 생성. 광고주명/소재명만 포함하고 목록 컨텍스트 배제

### I-7. 모바일 알림 채널 부재 — 외근 빈도 높은 AE 대응 불가 [상]
- **현상**: AE는 광고주 미팅, 현장 확인 등으로 외근 빈도 높음. 인앱 알림은 CMS 접속 시에만, 브라우저 푸시는 노트북 닫으면 미수신. 검수 실패, 매체 장애 등 긴급 상황에서 알림 도달 실패
- **문제**: 06 알림 관리의 3개 채널(인앱, 브라우저 푸시, Slack) 모두 모바일 커버리지 없음. 김서영 인터뷰(매체사)에서는 Slack 모바일 앱으로 우회 가능했으나, 영업대행사는 Slack 설정 권한 없음
- **관련**: 06 이벤트 카탈로그 채널 구성, 06 미결 "이메일 알림 채널 추가 여부"
- **제안**: 긴급 우선순위 이벤트에 대해 SMS 또는 카카오 알림톡 채널 추가 검토. 최소한 영업대행사의 검수 실패 + 매체 오프라인 이벤트에 대해 모바일 도달 보장 필요

## 추가 리서치 필요 영역

- **검수완료→편성→송출 상태 연계 표시 설계** — 소재 도메인 내에서 편성 상태를 어디까지 노출할지 (매체명만? 편성 기간까지? 송출 시간대까지?) 상세 수준 확정 필요. 편성 시나리오(05) 담당자와 교차 검토
- **매체별 해상도 마스터 데이터 존재 여부** — UC-14 검수에서 매체 실제 해상도와 교차 검증하려면 매체 등록 시 해상도 필수 입력 필요. 02 매체 관리 시나리오 확인
- **SMS/카카오 알림톡 채널 기술적·비용적 실현 가능성** — 인앱+푸시+Slack 외 채널 추가 시 06 시나리오 전반에 영향. 발송 비용(건당 과금), 발송 대상 관리(전화번호 수집) 등 검토
- **소재 버전 관리 스토리지 비용 영향** — 교체 이력 보존 시 스토리지 사용량 증가. 영상 파일 특성상 버전당 수백 MB 가능. 보존 버전 수/기간 제한 정책 필요
- **영업대행사 복수 광고주 관리 시 소재 목록 분리 UX** — 40~50개 소재를 광고주별로 구분하는 필터/폴더 구조 상세 설계. 현재 MAT-BR-07은 매체별 폴더만 정의
