## 1. 진단 — 왜 공장형으로 보이는가

개별 토큰이나 컴포넌트가 잘못된 게 아니다. 문제는 **"올바른 기본값"만으로 전체 화면을 조립**하면, 모든 화면이 같은 리듬, 같은 깊이, 같은 온도를 가지게 된다는 것이다.

이것은 디자인 시스템의 실패가 아니라, **디자인 시스템을 "적용"만 하고 "해석"하지 않은 결과**다. 시스템은 악보이고, 각 화면은 연주다. 같은 악보라도 곡마다 강약과 템포가 달라야 한다.

---

## 2. 증상과 근본 원인

### 7가지 증상

| # | 증상 | 설명 |
|---|------|------|
| 1 | 균일한 리듬 | 모든 페이지가 헤더→필터→테이블 같은 구조 |
| 2 | 컬러 단조 | primary green + warm gray 2톤으로만 구성 |
| 3 | 평탄한 위계 | 모든 정보가 같은 크기, 같은 weight |
| 4 | 범용 컴포넌트 | 도메인 특화 UI 없이 card+table 반복 |
| 5 | 깊이 획일 | shadow, border가 전부 동일 elevation |
| 6 | 기계적 간격 | 4px 그리드를 의도 없이 기계적 적용 |
| 7 | 브랜드 부재 | 어떤 B2B SaaS에든 붙일 수 있는 범용 껍데기 |

### 3가지 근본 원인

**컨텍스트 무시** (→ 증상 1, 4)
모든 화면에 같은 레이아웃 공식 적용. 대시보드(모니터링)와 목록(탐색)과 상세(집중)가 같은 구조를 가짐.

**시각적 에너지 부재** (→ 증상 2, 3, 5, 6)
컬러, 깊이, 스페이싱 모두 "안전한 기본값". 의도적으로 강조하거나 후퇴시키는 조절이 없음.

**제품 정체성 공백** (→ 증상 7)
"DOOH 옥외광고 관리"라는 도메인의 고유한 시각 언어가 없음.

## 3. 탈출 원칙

### 원칙 1. 화면은 역할이 다르면 생김새도 달라야 한다

**왜:** 대시보드는 "한눈에 상황 파악", 목록은 "원하는 것 찾기", 상세는 "깊이 있게 확인". 목적이 다른데 같은 레이아웃이면 사용자가 "지금 어디에 있는지" 체감하기 어렵다.

**대신:** 화면 유형별(모니터링 / 탐색 / 편집 / 집중열람) 레이아웃 변주를 허용한다.

**판단 기준:** "이 화면의 스크린샷만 보고 어떤 페이지인지 구분할 수 있는가?"

### 원칙 2. 모든 정보가 같은 볼륨이면, 아무것도 들리지 않는다

**왜:** stat-card 4개가 같은 크기, 테이블 행이 같은 높이면 시선이 갈 곳이 없다. "전부 중요"는 "전부 안 중요"와 같다.

**대신:** 화면마다 "주인공 1, 조연 2~3, 배경 나머지"를 정하고 크기·색·위치로 차등을 둔다.

**판단 기준:** "이 화면에서 사용자가 처음 3초간 봐야 할 것은 무엇인가?" 그것이 시각적으로 가장 크거나 강한가?

### 원칙 3. 컬러는 장식이 아니라 의미를 전달하는 도구다

**왜:** primary green이 버튼에도, 뱃지에도, 사이드바에도, 숫자에도 쓰이면 "초록 = 무엇"인지 의미가 희석된다. 반대로 화면 전체가 무채색이면 시선 안내 수단이 사라진다.

**대신:** primary는 "행동을 유도하는 곳"에만. 도메인별 accent color 도입 시 design-foundation.md에 스케일 등록 필수. 상태색은 상태에만.

**판단 기준:** "이 화면에서 primary color가 나타나는 곳을 모두 세어라. 행동 유도 외 용도가 있는가?"

### 원칙 4. 깊이와 간격으로 구조를 말해야 한다

**왜:** 같은 shadow, 같은 border, 같은 padding이면 "이 카드와 저 카드의 관계"를 읽을 수 없다. 그룹핑, 종속, 병렬 — 정보의 관계가 시각적으로 표현되지 않는다.

**대신:** elevation을 3단계로 구분(바닥/표면/부유). 스페이싱 토큰을 의도적으로 사용 — 모든 곳에 같은 값을 넣지 않고, 섹션 간 간격과 내부 간격의 비율 차이로 구조를 표현. 관련 없는 요소 사이에 의도적 빈 공간(breathing room).

**판단 기준:** "카드의 border를 전부 지워도, 그룹핑이 읽히는가?"

### 원칙 5. 도메인이 곧 개성이다

**왜:** DOOH는 "물리적 스크린이 도시에 존재하는" 사업. 매체는 위치가 있고, 편성은 시간축이 있고, 소재는 영상이다. 이 도메인 고유의 시각 요소(지도, 타임라인, 썸네일)가 화면의 주인공이 되어야 범용 SaaS와 차별화된다.

**대신:** 각 도메인의 핵심 시각 요소를 식별하고, 그것이 화면의 중심에 오도록 설계한다. 테이블은 보조 수단이지 메인이 아니다.

**판단 기준:** "도메인 특화 UI가 화면의 시각적 중심(focal point)인가?" 모니터링/집중열람형은 면적 50% 이상, 탐색형(목록)은 30% 이상을 지향.

### 원칙 6. 표면에 시각적 에너지가 있어야 한다

**왜:** 구조(레이아웃, 위계)를 아무리 바꿔도 표면(색, 질감, 시각 요소)이 "warm gray + white + 얇은 shadow"이면 같은 공장 제품으로 보인다. AI가 만드는 어드민 대시보드의 기본 레시피 자체를 깨야 한다.

**대신:** 세 가지 수단 중 최소 하나를 화면에 적용한다:
1. **컬러 블록** — 사이드바, 히어로 카드 등 큰 면적에 짙은 색(다크 그린, 다크 네이비 등)을 과감하게 사용
2. **데이터 시각화** — 숫자를 도넛, 바 차트, 프로그레스 바, 스파크라인 등으로 시각화. 단, "이 시각화를 제거하면 정보 전달이 나빠지는가?"가 Yes인 경우에만
3. **시각 요소 다양성** — 한 화면에 카드+숫자+텍스트만 반복하지 않고, 최소 2가지 이상의 다른 시각 표현(차트, 지도, 타임라인, 썸네일 등)을 포함

**판단 기준:** "이 화면의 스크린샷에서 텍스트를 전부 지워도 시각적으로 흥미로운가?" 직사각형 카드와 직선만 남는다면 에너지 부족.

## 4. 적용 가이드

### 화면 명세(GATE D1) 단계 체크

화면 명세 작성 시 다음 질문에 먼저 답한다:

1. **이 화면의 유형은?** (모니터링 / 탐색 / 편집 / 집중열람 / 기타)
2. **주인공은 무엇인가?** (사용자가 3초 안에 봐야 할 핵심 정보)
3. **도메인 특화 UI가 있는가?** (지도, 타임라인, 미리보기, 차트 등)
4. **이전에 만든 화면과 구조가 같아지려 하는가?** 같다면 의도적인가?
5. **숫자만 있는 카드가 3개 이상인가?** 그렇다면 시각화(도넛, 바, 프로그레스)로 대체 검토
6. **화면에서 텍스트를 지우면 시각적으로 흥미로운가?** 아니라면 컬러 블록이나 시각 요소 추가

### 5 활력 증후군 체크 (구현 완료 후)

구현 완료 시 `design-evaluator`가 자동으로 검사한다. 수동 검토 시 아래 기준을 사용:
 
- [ ] **S1 균일 여백**: 주인공 요소의 padding이 일반 요소보다 50% 이상 넓은가?
- [ ] **S2 카드 공장**: 통계/목록/알림 등 독립 객체가 아닌 것에 카드를 씌우지 않았는가?
- [ ] **S3 인터랙션**: hover + active + focus-visible 3가지 상태가 모두 구현되고, 물리적 이동(translateY)이 있는가?
- [ ] **S4 타이포**: font-weight 3단계 이상, 큰 숫자에 letter-spacing ≤ -0.02em이 적용되었는가?
- [ ] **S5 경계선**: 활성/선택 상태를 border-color 변경이 아닌 background tint로 표현했는가?
 
처방 CSS 패턴은 `.claude/skills/frontend-design-impeccable/reference/visual-vitality.md` 참조.
 
### 자동 검증
 
5 활력 증후군 체크는 `design-evaluator` 에이전트의 Step 4 (AI Slop Checklist)에 내재화되어 있다. `/design-review` 실행 시 자동으로 S1–S5 항목이 검사되어 리포트에 포함된다.

원칙의 자동 검증(critique 축 추가, 전용 에이전트 등)은 다음 프로토타입을 만들면서 "어디서 걸렸으면 좋았겠다"를 체감한 뒤 방식을 결정한다.
