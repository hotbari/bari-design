# 회의 1

포도 CMS(콘텐츠 관리 시스템)의 운영 현황과 기능을 검토하기 위해 진행된 실무 미팅의 내용을 정리한 것이다. 포도 측 운영 담당자가 실제 운영 중인 시스템을 기반으로 기능을 시연하고 설명하였으며, 개발팀은 향후 네이버 연동 CMS 구축을 위한 요구사항과 개선 방향을 파악하는 것을 목적으로 참여하였다.

## 사용자 구조 및 권한 체계

CMS의 사용자는 크게 어드민, 매체사, 광고주 세 유형으로 구분된다. 어드민은 포도가 단독으로 보유하며 모든 매체를 조회·관리할 수 있다. 매체사는 자신이 등록한 매체만 접근 가능하며, 실질적으로 CMS를 운영하는 주체이다.

매체사란 전광판(옥외 광고 매체)을 소유하거나 운영하는 주체로, 건물주·임차인·운영 대행사(포도 포함) 모두 해당된다. 광고주 계정은 자신의 계약 건에 대한 레포트 열람 용도로 개발되었으나 실제 사용 사례가 거의 없어 신규 CMS에서는 제거하기로 결정하였다. 광고주 관련 업무는 매체사 내부 프로세스(엑셀 편성표 등)로 처리되고 있다.

사용자 등록 순서는 매체사 등록 → 해당 매체사의 사용자 계정 생성 순이며, 사용자별 권한(전체/일부)을 설정할 수 있으나 현재는 대부분 전체 권한으로 운영 중이다.

## API 연동 및 광고 서버 구조

현재 CMS는 네이버 애드부스트, 하이브스택, 비스타 등 복수의 SSP(Supply-Side Platform) 광고 서버와 API로 연결되어 있다. 신규 CMS에서는 1차적으로 네이버 SSP 연동만 구현하고, 나머지 SSP는 추후 확장 가능한 구조로 설계하기로 하였다. 네이버 애드부스트는 광고대행사가 사용하는 별도 시스템으로, 신규 CMS와는 직접 관련이 없으며 데이터 송수신 연결 대상으로만 이해하면 된다.

## 매체(디바이스) 등록 및 관리

매체 등록 시 입력 항목은 디바이스명, 화면 사이즈(Top·Left·높이·너비), 서비스 시작·종료 시간, 위도·경도, SSP 연동 ID 등이다. 구좌 및 기준 초 항목은 실사용이 없어 신규 CMS에서 제거한다. 유동인구 연동 항목은 별도 논의가 필요하다.

- Top·Left 값: 전광판이 연결된 PC의 화면 내 출력 위치를 픽셀 단위로 수동 조정하는 값으로, 현장 방문 후 직접 확인하여 입력한다.
- 위치 고정: 조정된 위치 값을 플레이어에 고정 적용하는 기능으로, 변경 사항은 캘린더 신호를 통해 플레이어에 전달된다.
- 사용 유보: 매체 운영 여부를 토글하여 불필요한 알람을 방지하는 기능이다.
- 콜 타임: 플레이어와 CMS 간 새벽(약 3시) 통신 시간으로, 편성표 수신 및 소재 다운로드를 처리하며 트래픽 집중 방지를 위해 랜덤으로 배정된다.
- 위경도: 대시보드 지도 표시 용도이며, 옥외 고정 전광판이므로 이동 매체는 고려하지 않는다.

현재 운영 중인 매체는 약 15~20개이며, 플레이어는 Windows 기반 EXE 파일로 제공된다(Linux 미사용).

## 구좌 개념

구좌란 전광판이 하루 운영 시간 동안 편성할 수 있는 광고 슬롯의 총량을 의미한다. 예를 들어 18시간(64,800초) 운영 시 20초 기준 100회 보장 조건이면 총 30구좌가 산출된다. 이 중 공익광고 의무 편성 비율(서울시 10% + 문체부 10% = 20%, 즉 6구좌)을 제외한 24구좌가 실제 판매 가능 슬롯이다. 구좌 개념은 스케줄 편성 및 소재 등록 화면에서 활용되므로 개발팀의 이해가 필수적이다.

## 유동인구 데이터

유동인구 데이터는 포도가 전광판 가시권을 직접 조사하고 SKP(SK플래닛)로부터 FTP를 통해 수신한 통신사 데이터를 가공하여 CMS에 업로드하는 방식으로 운영된다. 이 데이터는 광고대행사가 매체 가치를 평가하고 광고 단가를 산정하는 기초 자료로 활용되며, 네이버 SSP의 RATP 필수 항목이기도 하다. 현재 수동 업로드 방식의 자동화 및 비용 문제(데이터 구매 비용이 상당함)에 대해 네이버와 별도 협의가 필요하다.

## 플레이어 모니터링 및 업데이트

현재 플레이어 이상 발생 시 슬랙 알림으로 담당자에게 통보되며, 원격 접속을 통해 수동으로 업데이트한다. 신규 CMS에서는 다음 항목의 강화를 검토한다.

- 플레이어 애플리케이션 상태, 네트워크 연결, CPU/GPU 온도, 메모리 등 하드웨어 정보를 플레이어가 수집하여 CMS로 전송
- 슬랙 알림 연동 유지(네이버도 슬랙 사용)
- 플레이어 자동 업데이트는 새벽 콜 타임에 수행하는 방향으로 검토하되, 매체사 실무자의 거부감을 고려하여 사전 고지 방식 병행

RFP에 명시된 CCTV 연동, 셋탑 온도·습도·도어 센서 등은 현재 운영 환경(데스크탑 기반)과 맞지 않아 실질적 구현 필요성이 낮으며, 네이버와의 사전 미팅을 통해 RFP 항목별 구현 범위를 조율하기로 하였다.

## 매체 그룹핑(사이니지 확장)

RFP에 매체 그룹핑 기능이 필수 항목으로 포함되어 있어 초기 설계에 반영하기로 하였다. 다만 사이니지 수준의 대규모 그룹 운영(수백~수천 대)은 1차 오픈 범위에서 제외하고, DB 설계 시 그룹 관리 테이블을 확장 가능한 구조로 준비하는 것으로 합의하였다. 매체사가 이미 그룹 단위로 매체를 묶어 영업하므로, 시스템은 매체 단위 관리를 기본으로 한다.

## 결론 및 다음 단계

- 제거 항목: 광고주 계정·생성 기능, 구좌·기준 초 입력 필드, 하이브스택·비스타 SSP 연동 메뉴(추후 확장 구조만 유지), CCTV·센서 관련 기능
- 유지·강화 항목: 매체사 등록·사용자 관리, 네이버 SSP API 연동, 플레이어 상태 모니터링(CPU/GPU 온도 등), 콜 타임 기반 편성 동기화, 매체 그룹핑(DB 설계 수준)
- 협의 필요 사항: 유동인구 데이터 자동화 및 비용 분담(네이버 협의), 플레이어 소스 및 API 가이드 제공(포도 개발 이사), 인프라 환경(클라우드, Docker/Kubernetes 등) 확정, 상주·비상주 개발 방식, 로고 및 제품명 결정
- 플레이어 테스트 환경: 포도 측이 Windows EXE 플레이어 파일을 재공유하며, 개발팀 사무실에 테스트 장비 1대 설치를 검토한다.
- 설계 정리 후 네이버와 사전 미팅을 통해 RFP 구현 범위를 공식 조율할 예정이다.

# 회의 2

---

### 1. 용어 및 이해관계자 정리

현업 담당자(A)가 **매체주**(전광판 소유자), **운영 대행사**(매체주 대신 편성·운영), **영업 대행사/광고주**(매체사에 광고를 의뢰하는 주체)의 역할을 명확히 구분하였다. 광고주가 직접 매체사에 연락하거나 대행사를 통해 연결되며, **네이버** 외에도 매체사가 자체 영업으로 광고를 수주하는 케이스가 다수 존재함을 확인했다.

### 2. 기존 전광판 시장 및 CMS 현황

국내 전광판 시장은 낙후되어 있으며, 기존 플레이어는 소재 등록·재생 기능만 제공한다. **포도**가 이를 개선하기 위해 현 CMS를 개발했으나, 이후 네이버가 자체 CMS 개발에 착수하면서 경쟁 구도가 형성되었다.

### 3. 계약 관리 기능

계약 관리 기능(광고주 등록 → 계약 생성 → 구좌·기간·가격·광고 유형 입력)은 **실무자들이 실제로 사용하지 않아** 유명무실한 상태다. 계약 시작·종료 시간은 편성과 연동되지 않고 단순 기록 용도에 그친다. 현업 담당자는 계약 단위 구좌 관리 대신 **편성표 기준으로 잔여 구좌를 자동 계산**하는 방식이 실무에 더 적합하다고 제안했다.

### 4. 소재 관리 기능

- 소재는 **매체별 폴더**로 구분 관리하며, 동일 광고라도 매체 해상도가 달라 소재 파일이 복수 존재한다(예: 신한은행용·용평용·비발디용).
- 파일 포맷은 **MP4 / H.264 / 30fps**로 단일화하여 인코딩 오류를 최소화하고 있다. 이미지 파일은 원칙적으로 받지 않으며, 불가피한 경우 광고주가 동영상으로 변환하여 제출한다.
- **파일 검수 시스템**: 소재 업로드 후 재생·캡처 자동 검증 → 정상이면 검수 여부 Y 자동 전환. 인코딩 오류로 화면이 하얗게 출력되는 사례가 실제 발생했으며, 이 경우 수동으로 검수를 N으로 초기화 후 재검수한다.
- **파일명 덮어쓰기**는 위험성으로 인해 비활성화되어 있다.
- 현재 소재는 파일 시스템으로만 관리되며, 만료 소재가 누적되어 정리가 어렵다는 문제가 있다.

### 5. 주요 개선 제안 (현업 담당자)

1. 소재 등록 시 **운영 기간·구좌 수 입력** 필드 추가 → 기간 만료 시 자동 표시 및 일괄 삭제 지원
2. 편성표 기반 **잔여 구좌 자동 계산** 기능으로 계약 관리 대체
3. 소재 목록에 **썸네일 보기**, 만료·진행 중·예정 **상태 필터** 제공
4. **다량 파일 일괄 등록** 기능 필수화(말일 편성 집중으로 인한 실무 부담 해소)
5. CMS 인코딩 변환 기능은 서버 부하로 **불필요**하다고 판단

### 6. 데이터 이관

포도 CMS에서 신규 CMS로 전환 시, 현행 편성표·소재 파일을 드라이브로 전달받아 수동 이관하는 방식을 사용해 왔다. 매체당 유효 소재는 약 30개 수준이며, 이관 작업은 현업 담당자가 협력하기로 했다.

## 2. 핵심 액션 아이템

1. **소재 관리 UI 전면 재설계**: 운영 기간·구좌 수·만료 상태 표시, 상태별 필터(진행 중/만료/예정), 썸네일 보기, 일괄 등록·삭제 기능을 포함하도록 설계한다.
2. **편성표 기반 잔여 구좌 자동 계산 기능 구현**: 계약 관리 기능을 대체하여 편성 시간 기준으로 사용·잔여 구좌를 실시간 산출한다.
3. **파일 포맷 업로드 시점 강제 검증 적용**: 업로드 단계에서 MP4 / H.264 / 30fps 외 포맷을 차단하고, 파일 검수(재생·캡처 자동 검증) 시나리오를 신규 UI에 맞게 재정의한다.
4. **계약 관리 기능 폐지 또는 최소화 검토**: 실사용률이 없으므로 공익광고·하우스 애드 등 필수 카테고리 분류만 잔존시키고 나머지는 제거한다.
5. **소재 파일 저장 구조를 오브젝트 스토리지(예: Minio)로 전환**: 현행 파일 시스템 기반 관리의 한계를 극복하고 이관·확장성을 확보한다.
6. **데이터 이관 프로세스 수립**: 포도 CMS의 편성표·소재 파일을 신규 CMS로 이관하는 절차를 문서화하고, 현업 담당자와 공동 검수 일정을 수립한다.
7. **다량 파일 일괄 등록 기능 필수 구현**: 말일 편성 집중 업무 특성을 반영하여 복수 파일 동시 업로드 및 일괄 검수 흐름을 지원한다.

3. **이미지 파일 수용 여부**(MP4 단일화 vs. 이미지 허용)를 팀 내에서 최종 결정 후 소재 가이드 문서에 반영.
4. **소재 자동 삭제 정책**(만료 후 일정 기간 경과 시 삭제 여부)은 재계약 케이스를 고려하여 추가 논의 필요.
5. 데이터 이관 일정은 신규 CMS 설계 완료 후 현업 담당자와 별도 협의.

# 회의 3

본 문서는 포도 CMS(Content Management System)의 광고 편성 기능 전반에 대한 실무 데모 및 기술 검토 미팅 내용을 정리한 것이다. 현업 운영자가 편성 구조, 슬롯 시스템, 캘린더 적용, 싱크 송출, 레포트, 그리고 NAM 연동 방식을 순서대로 시연하며 설명하였고, 개발·기획 담당자들이 질의응답을 통해 시스템 이해도를 높이는 방식으로 진행되었다.

---

## 편성표(재생 목록) 구조와 우선순위 시스템

편성표는 광고 소재가 언제, 어느 매체로 송출될지를 정의하는 '설계도'다. 편성 생성 시 시작 일시를 입력하지만, 이는 참고용이며 해당 날짜에 반드시 송출된다는 의미는 아니다.

편성에는 **1~3순위 우선순위 시스템**이 적용된다. 기본 편성은 3순위로 설정하고, 특정 날짜에만 다른 편성을 덮어씌울 때 2순위 또는 1순위를 사용한다. 운영 관리 보고용 촬영 시에는 촬영용 편성표를 별도로 만들어 1순위로 지정함으로써, 해당 시간대에 특정 광고 소재 1~2개만 집중 송출하여 촬영 대기 시간을 최소화한다. 예를 들어 반구좌 계약 소재는 풀구좌 기준 20분 사이클에서 40분에 한 번 노출되므로, 촬영 없이 대기하면 혹한기에 40분을 기다려야 하는 상황이 발생한다.

동순위 편성이 겹칠 경우 먼저 등록된 편성이 우선 적용되며, 동순위 상황은 가급적 방지하도록 권고하였다. 편성 수정 후에는 반드시 **즉시 반영** 버튼을 눌러야 플레이어에 전달되며, 이를 누르지 않으면 수정 내용이 송출에 반영되지 않는다.

---

## 슬롯 시스템과 구좌 계약 구현

편성표 생성 후 슬롯(폴더)을 채워 광고 소재를 배치한다. 자동 슬롯 만들기 기능은 최초 생성 시에만 사용 가능하며, 30개 슬롯 생성 시 UI 버그로 인해 31을 입력해야 30개가 생성되는 문제가 있다(미수정 상태).

슬롯 시스템의 핵심은 **분할 구좌 계약 구현**이다. 명동 신한은행 기준 20초 소재 1구좌는 1일 100회 이상 송출 조건으로 800만 원에 계약된다. 광고 예산 잔액을 소진하려는 광고대행사 수요로 반구좌·1/3구좌·1/4구좌·1/5구좌 계약도 발생한다.

- **1구좌**: 슬롯 1~30번 모두 소재로 채움 → 매 사이클마다 1회 노출
- **반구좌**: 슬롯 1번에 소재, 2번에 더미(스킵) 삽입 → 2사이클에 1회 노출
- **1/3구좌**: 더미를 1개 추가하여 3사이클에 1회 노출
- **1/4, 1/5구좌**: 동일 방식으로 더미 추가

기존 CMS에서는 반구좌 처리를 위해 편성표를 29번 슬롯 기준 58번까지 수동으로 늘려야 했으나, 이 폴더(슬롯) 시스템을 통해 편성표 행 수를 최소화하도록 기획되었다. 슬롯 내 소재 순서는 드래그로 변경 가능하다.

---

## 애드 슬롯과 외부 광고 플랫폼 연동

편성표 내 특정 슬롯은 **애드 슬롯**으로 지정하여 외부 광고 플랫폼(NAM, Hivestack, Vista)과 연동한다. 운영자는 가독성을 위해 애드 슬롯 번호를 100번대(Hivestack), 200번대(Vista), 300번대(네이버 NAM)로 구분하여 명명한다.

애드 슬롯이 편성 순서에 도달하면 플레이어가 해당 플랫폼과 통신하여 광고 소재를 수신한다. 광고가 없으면 스킵하고 다음 슬롯으로 넘어간다. 네이버 NAM의 경우 'N' 대신 'A'로 표기하는데, 이는 'N'이 다른 시스템 코드와 충돌하기 때문이다.

**데이터 흐름 정리:**

- CMS → 플레이어: 편성 정보 전송
- NAM → 플레이어: 광고 소재 직접 전송 (CMS를 경유하지 않음)
- NAM은 편성 정보를 보유하지 않으며, CMS와 NAM 간 직접 데이터 교환은 없음
- 플레이어는 콜 타임 신호를 기반으로 프리 캐싱을 통해 소재를 미리 수신

---

## 캘린더 적용 및 편성 복제

캘린더는 완성된 편성표를 실제 송출 기간에 배정하는 기능이다. 매체별 운영 시간(예: 오전 6시~자정)이 등록되어 있으며, 운영 시간 외 구간은 블랙(오프) 상태로 유지된다. 자정(00시) 표기 시 날짜 혼동이 발생하므로 23:59 방식으로 표기한다.

편성 복제 기능은 기존 편성표(예: 23일)를 다른 날짜(예: 28일)로 복사한 뒤 변경 소재만 수정하는 방식으로, 매번 처음부터 편성표를 작성하는 수고를 줄인다. 엑셀 템플릿 덤프 방식은 물리적 파일명 처리 문제로 채택하지 않았다. 편성표 익스포트(다운로드) 기능은 재생 목록과 편성표 대조 확인 용도로 필수 기능으로 분류되었으며, 복사 기능과 함께 반드시 포함되어야 한다.

---

## 싱크 송출(반복 편성)

싱크 송출은 매시 정각 등 지정된 시각에 특정 광고를 강제 삽입하는 기능이다. 홍대 인근 7개 매체가 동시에 동일 광고를 송출하는 방식으로 활용되며, 현재 운영 중인 매체 중 하나가 이 방식으로 운영된다.

설정 시 반복 시간(시·분·초 단위)을 지정하면, 해당 시각에 진행 중인 광고를 중단하고 싱크 광고를 우선 송출한다. 싱크 광고가 끝나면 중단된 광고는 로그에서 미완료 처리되고, 다음 광고가 처음부터 재생된다. 싱크 광고는 30구좌 운영 시 발생하는 약 1구좌 분량의 여유 시간을 활용하며, 기존 광고주의 계약 횟수(1일 100회 이상)는 별도로 보장된다.

초(second) 단위 설정은 초기 기획에 없었으나, 케이팝 콘텐츠 연계(예: 신라 스테이 매체의 4시 13초 요청), 아이돌 생일 기념 광고, 넷플릭스 등 영화 프로모션 연계 수요로 인해 추가되었다.

**긴급 편성**은 모든 조건을 무시하고 지정 소재만 송출하는 최고 우선순위 기능으로, 국가 재난 방송이나 서울시 공문에 의한 소등 캠페인(지구의 날 블랙 화면 송출 등) 대응 목적으로 설계되었다. 실제 사용 사례는 거의 없으나 유지 필요성이 인정되었다. 반복(싱크) 편성이 긴급보다 우선순위가 높으므로, 긴급 상황에서 싱크 광고가 끼어드는 문제를 방지하기 위해 별도 긴급 등급을 두었다.

---

## 레포트 및 시스템 성능

레포트 기능을 통해 특정 날짜·광고주별 송출 횟수와 내역을 조회할 수 있다. 데모 중 레포트 로딩이 비정상적으로 느려지는 현상이 발생하였으며, DB 연산 부하가 원인으로 추정되었다. 계약별 보고서 및 매체별 로그 조회 기능도 존재한다. 레포트 내용이 비어 있는 경우는 플레이어가 가동되지 않아 데이터가 없기 때문이며, 원격 세팅 지원을 통해 해결 가능하다.

## 날씨 연동 광고 및 RFP 관련 논의

날씨 조건 기반 광고 송출은 Hivestack 플랫폼의 캠페인 설정 기능을 통해 구현되며, 네이버 NAM에는 해당 기능이 없다. 따라서 현재 구축 범위에서는 날씨 연동 기능이 포함되지 않으며, 필요 시 네이버 측에 기능 추가를 요청해야 한다.

RFP상의 '광고 캠페인 부킹 등록' 항목은 CMS 기능과 광고 플랫폼(애드부스트) 기능이 혼재되어 작성된 것으로, 실제 구현 범위는 네이버로부터 캠페인 정보를 수신하여 목록으로 표시·등록하는 수준으로 정리되었다. 광고주가 직접 캠페인을 등록하는 기능은 CMS가 아닌 애드부스트 영역이다.
