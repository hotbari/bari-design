# 바리 챗봇 설계 명세

**작성일:** 2026-04-07
**상태:** 확정

---

## 1. 개요

사내 PDF 문서를 기반으로 직원들의 업무 질문에 답변하는 RAG 챗봇.
핵심 가치: **"이 챗봇으로 인해 내가 유능해졌다"** — 사용자가 더 빠르고 정확하게 업무를 처리하는 느낌을 줌.

**서비스명:** 바리
**사용 대상:** 사내 직원 (JAVA 대시보드 사용자)
**통합 방식:** 기존 JAVA 대시보드에 iframe embed (`?embed=true` — Chainlit 내장 기능, 헤더/툴바 자동 숨김. CORS 설정 필요 시 `CORS_ORIGINS` 환경변수로 허용 도메인 지정)

---

## 2. 사용자 역할

| 역할 | 권한 |
|------|------|
| 일반 직원 | 챗봇 사용 (질문/답변) |
| 관리자 | 챗봇 사용 + 문서 업로드/삭제 |

**인증 방식:**
- 챗봇 자체 로그인 없음. 모든 직원은 대시보드 iframe으로 접근.
- 문서 관리 패널은 별도 관리자 비밀번호로 보호. "문서 관리" 버튼 클릭 시 비밀번호 입력 모달 표시.
- 관리자 비밀번호는 환경변수 `ADMIN_PASSWORD`로 관리. 일치 시 세션 내에서 관리자 모드 유지.

## 3. 화면 구성

### 3.1 채팅 화면 (메인)

- 심플 센터 레이아웃
- 헤더: 바리 로고 + "문서 관리" 버튼 (항상 표시, 클릭 시 비밀번호 요구)
- 채팅 영역: 사용자 메시지 / 봇 답변 / 후속 질문 칩
- 하단 입력창

### 3.2 문서 관리 패널 (비밀번호 인증 후)

- "문서 관리" 버튼 클릭 → 비밀번호 모달 → 인증 성공 시 채팅 위에 오버레이 패널 슬라이드 업
- 구성: PDF 업로드 영역 (드래그 앤 드롭 + 파일 선택 버튼)만. 문서 목록/삭제 UI 없음.

**업로드 플로우:**
1. PDF 선택
2. 파일명 기준으로 Qdrant 기존 문서 유사 검색
3. 유사 문서 없음 → 바로 업로드 진행
4. 유사 문서 있음 → 목록 표시 후 선택 요구
   - "기존 문서 업데이트" 선택 → 해당 doc_id 청크 삭제 후 신규 저장
   - "신규 문서로 추가" 선택 → 그대로 추가
5. 업로드 중 Chainlit 기본 스피너 표시, 완료 시 성공 메시지

문서 삭제는 운영 레벨에서 Qdrant 콘솔로 직접 처리 (개발 범위 외)

## 4. 핵심 UX 피처

### 4.1 적응형 응답 방식

질문 유형을 LLM이 자동 감지하여 응답 방식 전환. 별도 UI 없음.

| 질문 유형 | 감지 기준 | 응답 방식 |
|-----------|-----------|-----------|
| 정보 조회 | "~이 뭐야", "~가 언제야" | 직답 1~3문장 + 출처 |
| 절차/방법 | "~하려는데", "~하는 방법" | 번호 목록 단계별 안내 |
| 검토 요청 | "~봐줘", "~검토해줘" | 항목별 분석 + 주의사항 선제 지적 |

시스템 프롬프트에 세 가지 응답 패턴과 전환 기준을 명시. LLM이 분류 후 해당 패턴으로 답변.

### 4.2 요약/상세 토글

- 구현 방식: LLM이 응답 생성 시 `[SUMMARY]...[/SUMMARY][DETAIL]...[/DETAIL]` 태그로 구분 → 스트림 완료 후 파싱하여 토글 UI 렌더링
- 기본값: 요약만 표시. "+ 상세 보기" 클릭 시 상세 내용 펼침 (LLM 추가 호출 없음)
- 노출 조건: 전체 응답 길이가 200자 이상일 때만 토글 표시. 미만은 전체 텍스트 그대로 표시.
- Fallback: 태그 파싱 실패 시 전체 텍스트 그대로 표시

### 4.3 출처 표시 (필수)

모든 답변에 출처 표시. 형식: `📎 문서명 p.XX`
문서에서 찾을 수 없는 내용은 LLM 자체 지식으로 답하지 않고 "업로드된 문서에서 관련 내용을 찾을 수 없습니다"로 거절.

### 4.4 후속 질문 제안

답변 아래 관련 질문 2~3개를 pill 형태로 제안.
LLM이 답변 생성 후 후속 질문을 별도 생성 (같은 호출에서 처리).
클릭 시 해당 질문 자동 입력 및 전송.

### 4.5 스트리밍 응답

OpenAI 스트리밍 API 사용. Chainlit의 `cl.Message.stream_token()` 으로 실시간 출력.

## 5. 기술 스택

| 레이어 | 기술 |
|--------|------|
| 챗봇 UI | Chainlit |
| 백엔드 | FastAPI (Chainlit 내장) |
| LLM | OpenAI `gpt-5.4-mini` |
| 임베딩 | OpenAI `text-embedding-3-small` |
| Vector DB | Qdrant (self-hosted 또는 Qdrant Cloud) |
| PDF 파싱 | pymupdf (fitz) |
| 검색 방식 | 하이브리드: Qdrant dense + sparse(FastEmbed BM25) 동시 저장, Qdrant 내장 RRF query fusion |

## 6. 데이터 플로우

### 6.0 빈 DB 초기 상태

문서가 하나도 없을 때 챗봇 첫 화면:
```
안녕하세요, 바리입니다.
사내 문서 기반으로 아래와 같은 것들을 도와드릴 수 있어요.

[📋 규정/정책 조회]  [🔢 업무 절차 안내]  [🔍 문서 검토]
```
pill 클릭 시 예시 질문 자동 입력.

### 6.1 문서 업로드 플로우

```
PDF 업로드
  → pymupdf로 텍스트 추출 (표·이미지 캡션 포함, 이미지 자체는 무시)
  → 1,000자(문자 기준) 청킹, 100자 오버랩
  → text-embedding-3-small으로 dense 벡터 생성
  → BM25 sparse 벡터 생성
  → Qdrant에 청크 저장
     payload: { source_filename, doc_id(UUID, 업로드 시 생성), page_number, chunk_index }
     업로드 전 파일명 유사도 검사 → 유사 문서 있으면 업데이트/신규 선택
     업데이트 선택 시: 기존 doc_id 청크 일괄 삭제 후 신규 저장
```

### 6.2 질문 응답 플로우

```
사용자 질문 입력
  → Qdrant 하이브리드 검색 (sparse + dense, RRF 병합, 상위 5청크 추출)
  → 시스템 프롬프트 + 세션 대화 히스토리 + 컨텍스트(5청크) + 질문 조합
  → gpt-5.4-mini 스트리밍 호출
  → [SUMMARY]/[DETAIL] 태그 파싱 → 토글 UI 렌더링
  → 출처 파싱 (source_filename + page_number) → 표시
  → 후속 질문 3개 생성 → pill 표시
```

## 7. 시스템 프롬프트 방향

- **역할:** 사내 문서 기반 전문 어시스턴트
- **말투:** 전문적이고 신뢰감 있는 한국어 (경어체)
- **규칙:**
  - 반드시 출처(문서명, 페이지) 포함
  - 문서에 없는 내용은 LLM 자체 지식으로 답하지 않고 거절 ("업로드된 문서에서 관련 내용을 찾을 수 없습니다")
  - 질문 유형(정보/절차/검토) 감지 후 해당 응답 패턴 적용
  - 답변을 `[SUMMARY]...[/SUMMARY][DETAIL]...[/DETAIL]` 구조로 작성
  - 답변 말미에 후속 질문 3개를 `[FOLLOWUP]Q1|Q2|Q3[/FOLLOWUP]` 형식으로 포함

## 8. 프로젝트 구조

```
bari-chatbot/
├── app.py                  # Chainlit 진입점, 메시지 핸들러
├── rag/
│   ├── pipeline.py         # 검색 + 프롬프트 조합 + LLM 호출 오케스트레이션
│   ├── ingest.py           # PDF 파싱, 청킹, 임베딩, Qdrant 저장/삭제
│   └── retriever.py        # Qdrant 하이브리드 검색
├── ui/
│   └── admin_panel.py      # 문서 관리 패널 UI (Chainlit 커스텀 액션)
├── config.py               # 환경변수 로딩, 전역 설정값
└── requirements.txt
```
